## 开场：一个真实的故事（3 分钟）

上个月，我用 AI 写了一个订单筛选功能。

AI 10 分钟就写完了，跑了一下，功能正常，提交代码。第二天 QA 测出一个 bug：筛选条件在 URL 参数同步时，特殊字符没做编码，导致页面白屏。

更让我后怕的是——我让 AI 自己检查，它说"代码没有问题"。

这件事让我意识到一个问题：**AI 写代码越来越快，但快不等于对。** 速度没有判断力，就是负债。

今天我想分享的，就是我在被 AI 坑了几次之后，总结出的一套工作方法——怎么让 AI 既快又靠谱。

## 一、AI 编程的现状：好消息和坏消息（3 分钟）

### 好消息

AI 编程已经不是实验阶段了，是行业基础设施：

- **92%** 的美国开发者每天在用 AI 编程工具（JetBrains 2025）
- **41%** 的全球代码由 AI 生成（JetBrains 2026）
- AI 编程市场规模：2025 年 47 亿美元 → 2027 年预计 123 亿

### 坏消息

但数据也揭示了另一面：

- **45%** 的 AI 生成代码包含安全漏洞（OWASP Top-10）
- AI 代码的**重复率是人工代码的 4 倍**（GitClear）
- **96%** 的开发者不完全信任 AI 生成的代码

这就是行业的"信任悖论"：96% 的人不完全信任，但都在用。82% 承认 AI 让他们更快，但只有 48% 总是在提交前 Review 代码。

### 概念的快速进化

2025-2026 年，业界对"怎么用 AI 写代码"经历了快速迭代：

```
Prompt Engineering (2023)     → 怎么措辞让 AI 回答更好
Vibe Coding (Feb 2025)        → 不管代码，用自然语言告诉 AI 你要什么
Context Engineering (Dec 2025) → 不只是措辞，而是组织正确的信息让 AI 理解
Agentic Engineering (Feb 2026) → 人来编排 Agent，Agent 来执行，保持专业质量
Harness Engineering (Feb 2026) → 设计约束系统，包裹 Agent 的完整控制
```

这些概念不需要都记住。你只需要记住一句话：

> **Agent = Model + Harness**
> 模型负责推理，Harness（约束系统）负责其他一切：上下文、约束、反馈循环、验证。

> You can outsource your typing to AI, but you can never outsource your understanding.
> —— Andrej Karpathy, 2026

**今天分享的核心就是：我为 AI Agent 设计了什么样的 Harness。**

## 二、我的工作流——用一个真实需求演示（15 分钟，核心）

先看整体框架，很简单，三个阶段：

```
Plan  →  Execute  →  Verify
（计划）   （执行）    （验证）
```

**两头是人，中间是 AI。** 人负责想清楚和验证对不对，AI 负责动手写。

下面我用一个真实需求走一遍完整流程。

> **案例：给订单列表页新增筛选功能**
> 需求：用户可以按状态、时间范围、关键词筛选订单

### 第一步：准备上下文（最重要的环节）

大部分人的用法是直接告诉 AI "帮我实现 XX 功能"。这就像把一个新人扔进项目，不给文档、不给介绍，直接让他写代码——写出来的东西肯定跑偏。

我每次给 AI 的信息分为两层：

**第一层：持久化上下文（一次写好，每次自动生效）**

我用 `CLAUDE.md` 文件给 AI 一个"项目入职手册"：

```markdown
# 项目规范

## 技术栈
- React 18 + TypeScript 5
- Zustand 状态管理
- Tailwind CSS

## 禁止事项
- 不要引入新的 UI 组件库，用项目内的组件
- 不要使用 any 类型
- 不要使用 localStorage，使用 Zustand persist

## 代码风格
- 函数组件 + Hooks
- 文件命名 kebab-case，组件命名 PascalCase
```

写好这个文件之后，AI 每次启动都自动知道项目规范，不用每次重复说明。**这比换一个更强的模型更能提升输出质量。**

> 比喻：持久化上下文就像"新人入职手册"——AI 不需要你每次都从头介绍项目。

**第二层：任务上下文（每次任务准备）**

```
我需要给订单列表页新增筛选功能。

## 需求
- 用户可以按状态、时间范围、关键词筛选订单
- 筛选条件要和 URL 参数同步（刷新不丢失）
- 移动端需要适配

## 约束条件
- 使用项目现有的组件库，不要引入新的
- 接口需要兼容现有的错误处理中间件
- 现有筛选逻辑用了 URL 参数同步，新增条件也要保持一致

请先阅读以下代码文件，理解现有架构，然后给我一个修改方案。
不要写代码，只告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 有没有需要注意的边界情况或风险
```

> 关键：**先出方案，不写代码。** 方案阶段修改成本接近零。

### 第二步：审方案（这是最关键的人工环节）

AI 会返回一个方案，大概长这样：

```
需要修改的文件：
- src/pages/OrderList/index.tsx — 新增筛选条件
- src/services/order.ts — 新增接口调用
- src/types/order.d.ts — 新增类型定义

修改思路：
- OrderList 页面：在现有筛选栏基础上新增下拉选择器
- order service：新增 fetchOrderFilter 接口
- 类型定义：补充 FilterOption 类型

注意事项：
- 现有筛选逻辑用了 URL 参数同步，新增条件也要保持一致
- 下拉数据需要做缓存
- 移动端需要考虑交互适配
```

**这一步，我会重点检查几件事：**

- **修改范围**：有没有漏文件？有没有改了不该改的？
- **业务正确性**：AI 不懂你的业务，可能技术正确但业务错误
- **安全意识**：45% 的 AI 代码有安全漏洞，不是开玩笑的
- **有没有更简单的方案**：AI 有时候会过度设计

> 开场那个故事里的 bug，就是因为我在这一步没仔细审——AI 没提到 URL 参数编码的问题，我也没注意到。

### 第三步：AI 写代码

方案确认后，让 AI 开始写：

```
方案确认，请按照上面的方案编写代码。
约束条件：
- 遵循项目现有代码风格
- 使用项目中已有的工具函数和组件
- TypeScript 类型要完整
- 不要引入新的第三方依赖
- 只修改方案中涉及的文件
```

**一个重要的心态转变：代码是可丢弃的。**

不满意？让它重写。方案走不通？换个方向再来。以前手写代码改一次成本很高，现在重新生成几乎零成本。这意味着你可以更频繁地尝试不同方案，而不是第一个方案就凑合用。

### 第四步：AI 写测试 + 自动修复

```
请为新增的代码编写单元测试。
重点：
- 覆盖正常流程
- 覆盖边界情况（空数据、异常输入、特殊字符）
- 覆盖异常处理（接口失败、loading/error 状态）
- 不要只测 happy path
```

现在的 Agent 工具支持自动修复循环：

```
AI 写测试 → 运行 → 失败 → AI 自动分析原因 → 自动修复 → 重新运行 → 通过
```

> 之前手动写单测是最容易被跳过的环节，现在基本都写了。**但要注意**：AI 生成的测试容易只覆盖 happy path，Mock 太多时测试通过但实际场景不一定对。要 Review 测试内容本身。

### 第五步：人工最终验证

这是最后一道防线：

| 检查项 | 关注点 |
|--------|--------|
| Code Review | 代码风格、重复逻辑、安全隐患、类型严谨性 |
| 功能验证 | 正常流程、边界情况、异常处理 |
| 体验检查 | 交互流畅度、loading 状态、错误提示、多端适配 |

> **标准不是"能不能跑"，而是"能不能长期维护"。**

### 流程回顾

```
准备上下文 → AI 出方案 → 人审方案 → AI 写代码 → AI 写测试 → 人验证
     ↑_______________ Harness（约束系统）________________↓
```

**AI 负责执行，人负责设计和验证。两头是人，中间是 AI。**

## 三、修 Bug 的不同做法：红测驱动（3 分钟）

开发新需求和修 Bug 是不同的工作模式。修 Bug 我用一套单独的流程：

> **先证明 bug 存在，再证明 bug 消失了。**

```
1. 把 bug 现象和相关代码给 AI
2. AI 写一个"失败测试"——能复现这个 bug（红灯）
3. 运行测试，确认确实失败 → 证明 bug 被复现
4. AI 修改代码修复 bug
5. 再跑测试，通过（变绿灯）→ 证明 bug 被修复
```

为什么这样做：

| 传统方式 | 红测驱动 |
|---------|---------|
| 凭经验猜原因 | 用测试精确定位问题 |
| 改完不确定修好没 | 测试通过 = 确认修复 |
| 可能引入新问题 | 测试持续 guarding，防止回归 |

## 四、效果和踩过的坑（5 分钟）

### 实际效果

- 需求开发效率：中等复杂度的需求，提速约 **40-50%**
- 单测覆盖率：从"基本不写"到"基本都写"
- Bug 率：没有明显上升（因为有人工 Review 兜底）

### 踩过的坑

**每一个坑，都帮我改进了一次 Harness。**

**坑 1：AI 会"自作聪明"**

有一次 AI 引入了一个项目里没人用过的状态管理库来解决问题。代码能跑，但其他人根本看不懂。

→ **Harness 改进**：在 CLAUDE.md 里明确写"禁止引入新的第三方依赖"，每次 Prompt 里再强调一次。

**坑 2：上下文不足导致方案跑偏**

AI 不了解业务背景，提出了一个技术上正确但业务上完全错误的方案——它建议删除一个看起来没用的字段，但那个字段是给财务报表用的。

→ **Harness 改进**：审方案时重点关注业务正确性，不只是技术实现。把关键业务规则写进 CLAUDE.md。

**坑 3：测试的"虚假安全感"**

AI 生成的测试全绿，但实际场景一测就崩——因为它 Mock 了所有外部依赖，测试的是 Mock 而不是真实逻辑。

→ **Harness 改进**：Review 测试内容本身，不只是看通过率。Prompt 里要求"Mock 最小化"。

**坑 4：代码重复**

AI 在三个不同文件里生成了几乎相同的工具函数，不知道项目里已经有现成的。

→ **Harness 改进**：在 CLAUDE.md 中列出已有的工具函数，让 AI 写代码前先搜索项目里有没有类似实现。

> **原则**（来自 Terraform 创始人 Mitchell Hashimoto）：
> 每次发现 Agent 犯错，都要花时间设计一个方案，确保它永远不会再犯同样的错。这就是 Harness Engineering 的核心。

## 五、给团队的建议（3 分钟）

一个扎心的数据：

> 80% 的企业在用 AI 编程工具，但只有 6% 有成熟的 AI 安全策略。

工具跑在治理前面。分享结束后，每个人可以立刻做的三件事：

**1. 今天就给自己的项目写一个 CLAUDE.md**

把技术栈、代码风格、禁止事项写进去。这是一次投入、持续收益的事情。模板我会发到群里。

**2. 下次让 AI 写代码前，先让它出方案**

不要直接写代码。方案阶段修改成本接近零，代码阶段修改成本高得多。

**3. AI 生成的代码，必须过 Code Review**

和人工代码同样的标准，甚至更严格。特别关注安全问题、重复代码、和业务正确性。

## 总结

AI 辅助开发有三个层次：

```
第一层：用 AI 写代码（工具层）——让 AI 帮你写函数、写页面
第二层：用 AI 管流程（方法层）——上下文工程、PEV 框架、红测驱动
第三层：设计 Harness（认知层）——从"写代码的人"变成"设计约束系统的人"
```

**核心公式：**

> Agent = Model + Harness
> 模型在飞速进化，但真正决定 AI 代码质量的，是你设计的 Harness。

**一句话概括：**

> 我不是在用 AI 写代码，我是在为 AI 设计一个可靠的运行环境。

### C. Prompt 模板

#### 模板 1：出方案

```
我需要实现 [功能描述]。

## 需求背景
- [背景说明]
- [用户场景]

## 约束条件
- [技术约束]
- [业务约束]

## 相关代码
- [文件路径 1]
- [文件路径 2]

请先阅读这些代码，理解现有架构，然后给出修改方案。
不要写代码，只需要告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 需要注意的边界情况和潜在风险
```

#### 模板 2：写代码

```
方案已确认，请按照方案编写代码。

约束条件：
- 遵循项目现有代码风格（参见 CLAUDE.md）
- 复用项目中已有的工具函数和组件
- TypeScript 类型要完整，不要使用 any
- 不要引入新的第三方依赖
- 只修改方案中涉及的文件
```

#### 模板 3：写测试

```
请为新增的代码编写单元测试。

测试要求：
- 覆盖正常流程
- 覆盖边界情况（空数据、异常输入、特殊字符）
- 覆盖异常处理（接口失败、loading/error 状态）
- 不要只测 happy path，重点测异常场景
- Mock 最小化，优先使用真实逻辑
```

#### 模板 4：修 Bug（红测驱动）

```
## 第一步：写红测

发现了一个 bug：
- 现象：[描述 bug 的具体表现]
- 复现步骤：[操作步骤]
- 期望行为：[应该是什么样]

相关代码：
[粘贴相关代码]

请先写一个测试用例来复现这个问题。
先不要修改业务代码。
```

```
## 第二步：修复

测试已经能复现问题了。请修改业务代码来修复这个 bug。
修复后，之前的测试应该能通过。
注意：不要修改测试用例来适配代码，而是修改代码来通过测试。
```

#### 模板 5：代码重构

```
这段代码需要重构，当前问题：
- [性能问题 / 可读性差 / 逻辑重复 / ...]

要求：
- 保持功能不变
- 提升可读性和可维护性
- 不引入新的第三方依赖
- 确保现有测试仍然通过
```
