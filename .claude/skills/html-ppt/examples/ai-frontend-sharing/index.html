开发工作流

进入第一部分——我的 AI 辅助开发工作流。

我用的是一个很简单的框架，叫PEV——Plan、Execute、Verify，三个阶段反复循环。跟传统开发的区别就一个核心原则：人做决策，AI 做执行。

这个框架不是我发明的，是我用了一年多 AI 辅助开发之后慢慢收敛出来的。接下来我拆开讲每个阶段具体做什么。

part 1 · framework

这是 PEV 框架的全景——Plan、Execute、Verify 三个阶段。

Plan 阶段，我会把需求描述、PRD 原文、约束条件一起喂给 Agent，但明确告诉它"先别写代码，只出方案"。方案出来后我仔细看完、确认无误才放行。确认后的方案会沉淀为 Spec，写进项目的 CLAUDE.md。

Execute 阶段，Agent 开始写代码和测试。约束条件给得很细——遵循代码风格、不能 mock、合理拆分组件、不要写兼容代码。写完代码立刻写测试，ESLint + TypeScript + Prettier + Vitest 四道防线不能破。

Verify 阶段是我的主场——Code Review、功能自测、体验检查。发现问题就进入 Fix 阶段。

最下面的虚线箭头代表反馈回路——发现质量不达标就回到 Plan，重新来过。这个循环不是一次性的。

part 1 · plan

PLAN：先想清楚，再动手

📥 告诉 Agent 三样东西

1. 需求描述（自己总结）
2. PRD 原文（完整参考）
3. 约束条件（能做/不能做）

🛑 关键操作

禁止 AI 写代码
"先不要写代码，只输出需求分析和实现方案。"

✅ Agent 方案要回答

· 实现什么功能？
· 改哪些文件？
· 影响哪些模块？
· 有没有风险点？

第二步不能省略：人工仔细看完方案，确认无误才放行。AI 可能误读 PRD，人的判断不可替代。

讲 Plan 阶段的三个动作。

第一步，告诉 Agent 三样东西。需求描述是我看完 PRD 后用自己的话总结的——这个动作很重要，因为它逼我自己先理解一遍。PRD 原文也发给 Agent 做参考，避免我的总结漏掉细节。约束条件就是边界——哪些能做、哪些不能改。

第二步是最关键的操作：禁止 AI 写代码。我会明确说"先不要写代码，只输出需求分析和实现方案"。为什么？因为方案阶段你需要的是思考，不是代码。

第三步，人仔细看完方案，确认无误才能放行。这一步不能被省略。

part 1 · plan

方案沉淀为 Spec，持久化到项目

🔄 跨会话复用

关掉终端再打开，Agent 还能读到之前的方案，不用从头沟通。

👥 团队协作

其他同学接手你的任务时，能看到完整的设计意图和决策背景。

🔍 可追溯

出了问题可以回溯"当初为什么这么设计"。

做法：维护 CLAUDE.md（或 AGENTS.md）— 技术栈、架构约定、代码风格、关键决策

核心理念：Spec 不是一次性输入，而是项目资产。2026 年行业趋势——Spec-Driven Development（规格驱动开发）。Andrew Ng & JetBrains 2026 专门出了课程推广。

Plan 阶段的第三步——方案沉淀为 Spec。

确认后的方案不要只留在对话记录里。我会把它写进项目的 CLAUDE.md 文件，提交到代码仓库。

这样做有三个好处：第一，跨会话复用——你关掉终端再打开，Agent 还能读到之前的方案。第二，团队协作——别人接手你的任务时能看到设计意图。第三，可追溯——出问题能回溯当初为什么这么设计。

这就是 2026 年行业说的 Spec-Driven Development——把需求规格当成和代码一样重要的工程制品，纳入版本控制。

part 1 · execute

EXECUTE：给定约束，让 AI 写代码

写代码时的约束

✅ 遵循当前项目的代码风格
✅ 不能有 mock 数据
✅ 合理拆分，单文件 ≤ 300 行
✅ 不要写兼容代码（不引入不必要的抽象）

约束越细 → AI 输出的代码越能直接用 → 返工越少

写完后立刻写测试

✅ 每个功能写完都要写单元测试
✅ 测试用例覆盖所有边界情况
✅ 每个测试用例都必须通过

四道防线：ESLint + TypeScript + Prettier + Vitest
代码不符合规范直接报错，AI 自动修复直到通过

Plan 确认后进入 Execute 阶段。

这时候给的约束会更具体。代码风格必须跟项目一致、不能出现 mock 数据、单文件不超过 300 行、不要写兼容代码。这些约束看起来细，但越细越好，因为约束越细，AI 输出的代码越能直接用，返工越少。

代码写完后紧接着写测试。四样——ESLint、TypeScript、Prettier、Vitest——叫前端项目的四道防线。代码不符合规范直接报错，AI 会在循环里自动修到通过为止。

AI 写代码能跑 ≠ 能上线。这一层人工验证是质量的底线。

代码写完了，测试也通过了，但我的工作还没完。Verify 阶段是我自己对 AI 产出做把关。

分两块来看。第一块是 Code Review。我检查三件事：代码风格和架构规范——AI 容易埋头写，不管项目里已有实现；安全隐患——硬编码密钥、缺少输入校验；还有类型定义和副作用。

第二块是功能自测。流程走一遍看完整性，检查交互体验，各种屏幕尺寸下布局是否正常。

AI 写代码能跑不等于能上线。这一层人工验证是质量的底线。

part 1 · verify 进阶

三重债务模型 — Martin Fowler & Storey, 2026

🛠️
技术债务
Technical Debt — "写得不够好"
紧耦合、缺错误处理、缺边界考虑。
AI 的 PR 比人类多 1.7 倍 重大问题。
Code Review 已经在查。
🧠
认知债务
Cognitive Debt — 理解在流失
代码不是你一行行写的，理解不够深。
团队变得不敢改代码，系统变成黑盒。
检查：我能给别人讲清楚吗？
🎯
意图债务
Intent Debt — "为什么"丢了
代码能表达做了什么，但很难表达为什么。
AI 写的代码尤其缺这个。
在关键决策处注释"为什么"，不是"做什么"。

应对：注释写"为什么"，Spec 里保留设计意图，代码就不容易变成只可执行不可理解的黑盒。

Martin Fowler 和 Margaret-Anne Storey 2026 年提了一个模型，叫三重债务模型，专门描述 AI 辅助开发带来的隐性成本。

第一重，技术债务——就是写得不够好。行业数据说 AI 生成的 PR 比人类多 1.7 倍的重大问题。这个我们在 Code Review 里已经在查了。

第二重，认知债务——代码不是你一行行写的，你对它的理解就不够深。应对就是 Verify 时问自己：我能不能给别人讲清楚这段代码为什么这么写？

第三重，意图债务——代码能表达做了什么，但很难表达为什么。AI 写的代码尤其缺这个。应对就是在关键决策处加注释，记录的是"为什么"。

如果自测发现 Bug，我的做法跟很多人不太一样——先让 Agent 写一个复现 Bug 的测试，再修代码。

流程就五步：发现 Bug、写一个失败的测试来复现问题、让 AI 修代码、跑测试确认通过、最终确认修复。这就是 TDD 的核心思路用在 debug 场景——先复现，再修复。

一句话理解 Agent：Agent = 大模型 + 工具 + 循环。

普通聊天机器人只能回复文字。你说"帮我查天气"，它从训练数据里翻一条出来告诉你——可能已经过期了。Agent 不同——它能做事。

看右边的例子：你说"帮我查天气"，Agent 会先思考、然后调用天气 API 去查、拿到结果再告诉你。它多了两个能力——调用工具和循环观察。

普通 LLM 像一个只读过书的人，Agent 像一个能上网、能操作软件、能执行命令的人。

part 2 · react

核心：ReAct = Reasoning + Acting

就是一个"边想边做"的循环

Agent 的工作方式叫 ReAct = Reasoning + Acting，就是一个"边想边做"的循环。

看这个三角形的循环图。最上面是思考——分析现在的状况，决定下一步做什么。左边是行动——调用工具来执行操作。右边是观察——拿到执行结果后评估：任务完成了吗？还需要继续吗？没完成就回到思考继续下一轮。

这个循环一直在转，直到任务完成或者达到次数上限。这就是为什么 Claude Code 能自己去读文件、改文件、修 lint——它在重复"思考→行动→观察"这个循环。

拿一个具体例子走一遍——加一个搜索框。

你在终端跟 Claude Code 说"帮我在用户列表页加一个搜索框"。

第一轮，Agent 思考：我得先知道这个页面现在长什么样。于是它调用 read_file 工具去读 UserList.tsx。读完观察到了 200 行代码，里面有表格组件。

第二轮，思考：基于现有代码，用 Ant Design 的 Input.Search 组件，在表格上方加搜索栏。

第三轮，思考：代码改完了，需要写测试。

第四轮，思考：lint 报了一个类型错误。行动：edit 修复。

第五轮，思考：功能实现、测试通过、lint 通过——任务完成。

五轮循环，一个搜索框就加好了。每一轮的决策你都能看清楚。它不是魔法，是可追踪的步骤序列。

part 2 · tech

三个关键技术点

🔧 工具 (Tools)

read_file · edit · bash · grep · glob
每个工具都有描述，告诉 AI "什么时候用、怎么用"。

📝 消息历史 (Memory)

维护完整消息记录。
每一轮都能看到前面的对话和操作结果。
所以能做多步操作不会"忘"。

🔄 循环上限

不是无限循环。
设置了最大迭代次数，防止"死循环"。
超了没完成就停下来让你介入。

带走两个认知：
1. AI 写代码不是魔法，是"思考→行动→观察"的循环
2. 人在环路上的位置：你定规则，AI 在规则内执行，最终你验证

三个关键技术点快速过一下。

第一个，工具。Agent 能用的工具不止打字。读文件、改文件、执行命令、搜索代码——每个工具都有描述告诉 AI 什么时候用、怎么用。

第二个，消息历史。Agent 维护一个完整的消息记录，每一轮思考它都能看到前面的所有对话和操作结果。这就像它有记忆一样。

第三个，循环上限。Agent 不是无限循环的，设置了最大迭代次数防止死循环。

这一部分的重点就两个认知：第一，AI 写代码不是魔法，是可理解的循环；第二，人在环路上的位置——你定规则，AI 在规则内执行，最终你验证。

part 3 · method

两步走

第一步：定期刷 GitHub Trending

每周花十几分钟扫一眼
github.com/trending
关注点：
· 解决什么问题？

不是为了追热点，是保持对"大家都在关注什么"的感知

第二步：AI 辅助读源码

传统方式：clone → 啃 README → 看目录 → 一点一点读 → 半天还没搞清核心
我的方式：
1. git clone 项目
2. 把项目目录喂给 AI Agent
3. 问 AI：
  "核心架构是什么？"
  "最关键的代码在哪？"
  "XX 功能怎么实现的？"
  "这个设计为什么这么写？"

为什么有效？读源码是最直接的学习方式，但门槛高。AI 帮你降低"看不懂"的门槛，相当于有一个随时可以问的导师。你是主动探索，不是被动输入——理解深度完全不一样。

学习方法就两步。

第一步，定期刷 GitHub Trending。每周花十几分钟扫一眼，看看最近最火的项目。不是追热点，而是保持对"大家都在关注什么"的感知。

第二步，AI 辅助读源码。传统方式是 clone 下来从 README 开始啃——半天过去了还没搞清楚核心逻辑。我的方式是把项目目录喂给 AI Agent，让它做代码导航和讲解。

为什么有效？读源码是最直接的学习方式，但门槛高——看不懂、找不着入口。AI 帮你降低了"看不懂"的门槛，你是在主动探索，不是在被动输入。

最后一部分聊一个想法——把我们的 SaaS 系统的业务能力暴露给 Agent，让它直接操作业务系统。这个想法不复杂，但我觉得值得团队讨论。

part 4 · idea

把业务操作变成 CLI 命令

我们已经有完整的 services/ 层和 API 接口，只需包一层 CLI

# 业务操作 CLI saas payment list --status=pending 查看待处理的付款单 saas payment detail --id=PAY20260509001 查看某笔付款详情 saas deal quote --pair=USD/CNH 查询实时汇率 saas deal execute --pair=USD/CNH --amount=10000 执行交易 saas approval list --pending 查看待审批列表 saas approval approve --id=APV001 审批通过 saas balance check --currency=USD 查询余额

本质：这些命令就是调用现有的 API 接口。CLI 只是加一层薄壳，技术上不难。

思路就是把常用的业务操作封装成 CLI 命令。

我们现在的系统，所有操作都通过网页界面完成。但如果把这些操作封装成 CLI 命令，你就可以在终端里用一条命令完成。

本质上这些命令就是调用现有的 API 接口。我们已经有完整的 services/ 层和后端 API，CLI 只是包一层薄壳。技术上不难。

part 4 · agent + cli

Agent 直接操作业务系统

# 场景一：日常查询 你: "查一下今天有多少笔待审批的付款" Agent: [执行 saas approval list --pending] 当前有 12 笔待审批付款，其中 3 笔超过 24 小时未处理 你: "把金额最大的那笔详情给我看" Agent: [执行 saas payment detail --id=PAY20260509087] 付款编号: PAY20260509087 金额: USD 500,000 收款人: ABC Trading Co., Ltd. 状态: 待审批 # 场景二：自动化流程 你: "帮我监控 USD/CNH 汇率，低于 7.20 时自动买入 10 万美元" Agent: 收到，开始监控... 14:32 汇率触及 7.198，已自动执行买入 交易编号: DEAL20260509001 成交价: 7.198 金额: USD 100,000

核心：让 Agent 成为业务系统的操作者。人定规则，Agent 在规则内自动执行。难的不是技术，是想清楚哪些操作适合暴露、权限怎么控制。

一旦业务操作变成了 CLI 命令，Agent 就能直接操作我们的业务系统。

看场景一。你在终端里用自然语言说"查一下今天有多少笔待审批的付款"，Agent 帮你执行命令然后告诉你结果。你不需要打开浏览器、登录系统、点点点。

更进一步的，看场景二——自动化流程。你说"帮我监控汇率，低于 7.20 时自动买入 10 万美元"，Agent 就在后台持续监控，触发条件时自动执行。

这个想法的核心是：让 Agent 成为业务系统的操作者。人定规则，Agent 在规则内自动执行。
