## 第一部分：我的 AI 辅助开发工作流（约 15 分钟）

### 整体框架：PEV

我用的是一个很简单的框架，三个阶段反复循环：

```
Plan  →  Execute  →  Verify
（计划）   （执行）    （验证）
```

```
┌──────────────────────────────────────────────────────────────────────┐
│               Plan → Execute → Verify（PEV 框架）                    │
│                                                                      │
│  ┌─ PLAN ──────────────────────────────────────────────────┐        │
│  │  1. 准备上下文 ──→ 2. AI 出方案 ──→ 3. 人工 Review 方案  │        │
│  └──────────────────────────────────┬───────────────────────┘        │
│                                     │ 确认无误                       │
│  ┌─ EXECUTE ────────────────────────↓───────────────────────┐       │
│  │  4. AI 写代码 ──→ 5. AI 写测试（含自动修复循环）           │       │
│  └──────────────────────────────────┬───────────────────────┘        │
│                                     │                                │
│  ┌─ VERIFY ─────────────────────────↓───────────────────────┐       │
│  │  6. 人工 Code Review + 功能验证 + 体验检查                 │       │
│  └──────────────────────────────────┬───────────────────────┘        │
│                                     ↓                                │
│                                  提测                          │
└──────────────────────────────────────────────────────────────────────┘
```

跟传统开发的区别就一个核心原则：**人做决策，AI 做执行**。

---

### PLAN 阶段：先想清楚，再动手

**第一步：我告诉 Agent 三样东西**

1. **需求描述**：我会先自己把 PRD 看完理解，然后用自己的话总结成一段文字告诉 Agent——这个需求做什么？最终目标是什么？
2. **产品文档（PRD）**：完整原文也发给 Agent 做参考，避免我的总结遗漏细节
3. **约束条件**：哪些能做，哪些不能做，哪些不能改

**关键操作：禁止 AI 写代码**

这一步我会明确告诉 Agent："先不要写代码，只输出需求分析和实现方案。"

Agent 输出的方案里需要回答几个问题：
- 要实现什么功能？
- 需要改哪些文件？
- 会影响哪些现有模块？
- 技术方案有没有风险点？

**第二步：我仔细看完方案，确认无误才放行**

这一步不能被省略。AI 可能会误读 PRD，可能会提出不符合项目架构的方案。人的判断在这里不可替代。

**第三步：方案沉淀为 Spec，持久化到项目里**

确认后的方案不要只留在对话记录里。我会把它写成一份 Spec 文档，提交到代码仓库。这样做有三个好处：

1. **跨会话复用**：你关掉终端再开，Agent 还能读到之前的方案，不用从头沟通
2. **团队协作**：其他同学接手你的任务时，能看到完整的设计意图和决策背景
3. **可追溯**：出了问题可以回溯"当初为什么这么设计"

具体做法：在项目根目录维护一个 `CLAUDE.md`（或 `AGENTS.md`）文件，里面记录：
- 项目的技术栈和架构约定
- 代码风格和目录结构规范
- 已确认的技术方案和关键决策

这就是 2026 年行业里说的 **Spec-Driven Development（规格驱动开发）**——把需求规格当成和代码一样重要的工程制品，纳入版本控制。Andrew Ng 和 JetBrains 在 2026 年专门出了课程推广这个做法。简单说：**spec 不是一次性输入，而是项目资产。**

> 📋 **案例模板**（此处填入实际案例，准备时按要素展开）：
> - **任务背景**：什么需求？涉及几个模块？原本预估多久？
> - **AI 方案输出**：AI 提出了什么方案？有没有你想调整的地方？
> - **方案调整**：你改了哪里？为什么？
> - **小结**：如果让人从零想方案要多久？AI 帮你省了多少时间？

---

### EXECUTE 阶段：给定约束，让 AI 写代码

方案确认后，开始让 AI 写代码。这时候给的约束条件会更具体：

- 遵循当前项目的代码风格
- 不能有 mock 数据
- 合理拆分代码，组件化开发（单文件不超过 300 行）
- 不要写兼容代码（不引入不必要的抽象）

约束给得越细，AI 输出的代码越能直接用，返工越少。

**代码写完，立刻写测试**

AI 写完代码后，紧接着让它写单元测试。约束条件：

- 每个功能写完都要写单元测试
- 每个测试用例覆盖所有边界情况
- 每个测试用例都必须通过

前端项目强校验防线：ESLint + TypeScript + Prettier + Vitest。代码不符合规范直接报错，AI 自动修复直到通过。

---

### VERIFY 阶段：人工把关，不能少

代码和测试都写完了，但我的工作还没完。我会逐项检查：

**Code Review：**
- 是否符合现有代码风格和架构规范
- 是否引入了重复逻辑（AI 容易埋头写，不管已有实现）
- 有没有安全隐患（硬编码密钥、缺少输入校验）
- 类型定义是否严谨
- 有没有潜在的副作用

**功能自测：**
- 整个流程走一遍，流程是否完整
- 交互是否流畅，有没有卡顿
- loading 状态展示是否合理
- 错误提示是否用户友好
- 各种屏幕尺寸下布局是否正常

---

### VERIFY 阶段要注意的隐性风险：三重债务模型

Martin Fowler 和加拿大计算机科学家 Margaret-Anne Storey 在 2026 年提出了一个模型，叫**三重债务模型（Triple Debt Model）**，专门描述 AI 辅助开发带来的隐性成本。了解这个模型，能帮你在 Verify 阶段更有针对性地检查。

**1. 技术债务（Technical Debt）—— 看得见的问题**

AI 生成的代码能跑，但可能写得不够干净：紧耦合、缺错误处理、缺边界情况考虑。行业数据显示，AI 生成的 PR 比人类写的多 1.7 倍的"重大问题"，主要是控制流错误和安全漏洞。

这个我们在 Code Review 里已经在检查了。

**2. 认知债务（Cognitive Debt）—— 团队的理解在流失**

这是更隐蔽的问题。当 Agent 大量生成代码后，团队成员可能"不知道系统为什么这么工作"了。代码不是你一行行写的，你对它的理解就不够深。久而久之，团队会变得不敢改代码——怕改了这里，那里出问题。系统变成一个**黑盒**。

应对方法：Verify 阶段不只是检查"代码对不对"，还要问自己"我能不能给别人讲清楚这段代码为什么这么写？"。讲不清楚的地方，说明认知债务在积累，需要花时间理解。

**3. 意图债务（Intent Debt）—— 丢失的"为什么"**

代码能表达"做了什么"，但很难表达"为什么这么做"。AI 写的代码尤其缺这个——它不会告诉你"我选了方案 A 而不是方案 B，是因为方案 B 在并发场景下有竞态问题"。

应对方法：在关键决策处加注释，记录的是"为什么"而不是"做了什么"。这也是前面说的 Spec 持久化的价值——spec 里保留了设计意图，代码就不容易变成"只可执行不可理解"的黑盒。

---

### FIX 阶段：TDD 驱动修 Bug

如果自测发现 Bug，我的做法是：**先让 Agent 写一个复现 Bug 的测试，再修代码。**

为什么这样做：
- 用测试精确定位问题，而不是凭感觉猜
- 改完代码不确定修好没？跑测试，过了就是过了
- 先复现，再修复——这是 TDD 的核心思路，只不过这里用在了 debug 场景

```
发现 Bug → 写一个失败的测试（复现）→ 让 AI 修代码 → 测试通过 → 确认修复
```

这部分就是这样一个闭环。人始终在环路里做决策，AI 负责帮你跑得快。

## 第二部分：AI Agent 的运行原理（约 15 分钟）

> 这一部分面向零基础的同学，不用技术背景也能听懂。

### 先看一个日常场景

你在终端跟 Claude Code 说："帮我在用户列表页加一个搜索框。"

然后你看到 Claude 自己去读文件、分析代码、写改动、写测试、跑 lint……最后告诉你做好了。

你有没有想过：**它到底是怎么做到这一串操作的？**

### 一句话理解 Agent

```
Agent = 大模型 + 工具 + 循环
```

```
┌─────────────────────────────────────────────────────────┐
│                        Agent                             │
│  ┌───────────┐    ┌───────────┐    ┌───────────┐       │
│  │   大模型   │ +  │   工具    │ +  │   循环    │       │
│  │  (思考)   │    │  (行动)   │    │  (观察)   │       │
│  └───────────┘    └───────────┘    └───────────┘       │
└─────────────────────────────────────────────────────────┘
```

普通聊天机器人只能回复文字。Agent 不同——它**能做事**。

举个例子：

```
普通 LLM：
  你: "帮我查北京天气"
  它: "北京今天晴，25度。"（这是训练数据里的知识，可能是过期的）

Agent：
  你: "帮我查北京天气"
  Agent：[思考] 我需要查询天气...
  Agent：[行动] 调天气 API 查
  Agent：[观察] API 返回：晴，25度
  Agent：[回答] 北京今天晴，25度。
```

---

### 核心：ReAct 原理

Agent 的工作方式叫 **ReAct = Reasoning（思考）+ Acting（行动）**

就是一个"边想边做"的循环：

| 阶段 | 做什么 | 对应到 Claude Code |
|------|--------|-------------------|
| **思考** | 分析现状，决定下一步 | Claude 判断"需要先看文件结构" |
| **行动** | 调用工具执行操作 | 调 `read_file` / `edit` / `bash` |
| **观察** | 拿到结果，评估是否继续 | 读到代码内容，判断还需不需要改 |

画成流程图：

```
用户输入任务
      ↓
  ┌─────────────────────┐
  │  思考：我要做什么？   │ ←──────────────┐
  └─────────┬───────────┘                  │
            ↓                              │
     需要调工具吗？                          │
     ↙        ↘                           │
   需要         不需要                       │
    ↓            ↓                         │
  选择工具     生成回答                       │
    ↓            ↓                         │
  执行工具     任务完成                    结果&继续思考
    ↓                                       ↑
  拿到结果 ─────────────────────────────────┘
```

**用加搜索框的例子走一遍：**

```
第 1 轮：
  思考：需要知道用户列表页长什么样
  行动：read_file("src/pages/UserList.tsx")
  观察：读到了 200 行代码，里面有表格组件

第 2 轮：
  思考：要用 Ant Design 的 Input.Search 组件，在表格上方加搜索栏
  行动：edit("src/pages/UserList.tsx", ...)
  观察：代码改完了

第 3 轮：
  思考：需要写测试
  行动：edit("src/pages/__tests__/UserList.test.tsx", ...)
  观察：测试写完了，跑一下

第 4 轮：
  思考：lint 报了一个类型错误
  行动：edit(...)
  观察：lint 通过了

第 5 轮：
  思考：功能已实现，测试通过，lint 通过，任务完成
  行动：无 → 直接回答"已完成"
```

这就解释了为什么 Claude Code 能自己去读文件、改文件、修 lint——它在重复 "思考 → 行动 → 观察" 这个循环，直到任务完成。

---

### 三个关键技术点（不深讲，有兴趣可以会后聊）

**1. 工具（Tools）**

Agent 能用的工具不止打字。Claude Code 的工具箱里有：
- `read_file`：读文件
- `edit`：改文件
- `bash`：执行命令（跑测试、装依赖、git 操作）
- `grep`：搜索代码
- `glob`：按文件名模式查找

每个工具都有描述，告诉 AI "什么时候用、怎么用"。

**2. 消息历史（记忆）**

Agent 维护一个完整的消息记录。每一轮思考它都能看到前面的所有对话和操作结果。这就是为什么它能做多步操作不会"忘"。

**3. 循环上限**

Agent 不是无限循环的。Claude Code 设置了最大迭代次数，防止"死循环"。如果超了还没完成，它会停下来让你介入。

---

### 这一部分小结

你不需要记住所有细节，但可以带走两个认知：

1. **AI 写代码不是魔法**，是"思考→行动→观察"的循环，你在流程图的每一轮都能看到它的决策
2. **人在环路上的位置**：你制定规则（约束条件），AI 在规则内循环执行，最终你验证结果

## 第三部分：我的 AI 学习方法（约 10 分钟）

> 这一部分很短，因为我的方法确实不复杂。

### 核心方法：GitHub Trending + AI 辅助读源码

**第一步：定期刷 GitHub Trending**

每周花十几分钟扫一眼 [github.com/trending](https://github.com/trending)，看最近最火的项目有哪些。不是为了追热点，而是保持对"大家都在关注什么"的感知。

关注点：
- 这个项目解决什么问题？
- 有没有新的开发范式或工具思路？
- 跟我现在的技术栈有没有关系？

**第二步：对感兴趣的项目，clone 下来 + AI 辅助读源码**

传统方式：clone 下来，从 README 开始啃，看目录结构，一点一点读源码，半天过去了还没搞清楚核心逻辑。

我的方式：

```
1. git clone <项目地址>
2. 把项目目录喂给 AI Agent
3. 问 AI：
   - "这个项目的核心架构是什么？"
   - "最关键的代码在哪个文件？"
   - "XX 功能是怎么实现的？"
   - "这个设计为什么这么写？有什么好处？"
4. AI 帮你做代码导航 + 讲解，你跟着它的分析读源码
```

> 📋 **案例**：我用这个方法读过 Claude Code 本身的源码。传统方式可能需要好几天，用 AI 辅助，几个小时就能理解它的 Agent 循环、工具系统、消息管理这些核心模块。

**为什么这个方法有效？**
- 读源码是最直接的学习方式，但门槛高（看不懂、找不着入口）
- AI 帮你降低了"看不懂"的门槛，相当于有一个随时可以问的导师
- 你是主动探索，不是被动输入——理解深度不一样



## 第四部分：团队可以尝试什么（约 5 分钟）

> 这一部分聊一个想法：把我们的 SaaS 系统的业务能力暴露给 Agent，让它直接操作系统。

### 思路：把业务操作变成 CLI 命令

我们现在的系统，所有操作都只能通过网页界面完成——查付款单、看交易记录、审批流程、查看余额，都要登录系统点来点去。

但如果我们把常用的业务操作封装成 CLI 命令，比如：

```bash
saas payment list --status=pending          # 查看待处理的付款单
saas payment detail --id=PAY20260509001     # 查看某笔付款详情
saas deal quote --pair=USD/CNH              # 查询实时汇率
saas deal execute --pair=USD/CNH --amount=10000 --direction=buy  # 执行交易
saas approval list --pending                # 查看待审批列表
saas approval approve --id=APV001           # 审批通过
saas balance check --currency=USD           # 查询余额
saas beneficiary list --customer=CUST001    # 查询收款人列表
```

这些命令本质上就是调用我们现有的 API 接口。我们已经有完整的 `services/` 层，后端 API 都在，只需要包一层 CLI。

### 这跟 Agent 有什么关系？

一旦业务操作变成了 CLI 命令，Agent 就能**直接操作我们的业务系统**。你不需要打开浏览器、登录系统、点点点，而是在终端里用自然语言告诉 Agent：

```
你: "查一下今天有多少笔待审批的付款"
Agent: [执行 saas approval list --pending]
       当前有 12 笔待审批付款，其中 3 笔超过 24 小时未处理

你: "把金额最大的那笔详情给我看"
Agent: [执行 saas payment detail --id=PAY20260509087]
       付款编号: PAY20260509087
       金额: USD 500,000
       收款人: ABC Trading Co., Ltd.
       状态: 待审批

你: "这笔交易的汇率是多少？"
Agent: [执行 saas deal quote --pair=USD/CNH]
       当前 USD/CNH 汇率: 7.2450
```

更进一步，Agent 可以做**自动化流程**：

```
你: "帮我监控 USD/CNH 汇率，当汇率低于 7.20 时自动买入 10 万美元"
Agent: 收到，开始监控... [后台持续运行]
       14:32 汇率触及 7.198，已自动执行买入
       交易编号: DEAL20260509001，成交价 7.198，金额 USD 100,000
```

或者：

```
你: "每天早上 9 点把昨天的交易汇总发给我"
Agent: [定时执行 saas report daily --date=yesterday]
       昨日交易汇总：
       总交易笔数: 47
       总交易金额: USD 2,340,000
       待处理: 5 笔
       异常标记: 1 笔（大额交易待复核）
```

### 这个想法的实质

让 Agent 成为业务系统的操作者。人定规则（什么时候买、审批条件是什么、什么情况需要通知），Agent 在规则内自动执行。
我们项目里已经有现成的 API 接口层（`services/`），CLI 只是加一层薄壳。这件事技术上不难，难的是想清楚哪些操作适合暴露给 Agent、权限怎么控制。但这些恰恰是值得团队讨论的。

> 📋 **建议下一步**：可以先挑一个最简单的场景做原型——比如把 payment list 做成 CLI 命令，让 Agent 能通过终端查询付款列表。验证通了这个思路，再扩展到更多业务操作。

4. **团队实践**：把 SaaS 业务操作封装成 CLI，让 Agent 直接操作系统

最后想说：AI 工具现在的进步速度比我们想象的快，但核心逻辑没有变——**它能放大你的能力，但不能替代你的判断**。会写约束条件、能判断方案好坏、能发现代码中的隐患——这些才是你应该持续打磨的。
