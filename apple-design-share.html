常见陷阱

为什么不能一上来就让 AI 直接写？

1

自信地写错

看起来没问题，实际上逻辑有误

2

重复造轮子

忽略已有方案，自己重写一遍

3

漏看上下文

只看到了局部，没理解全局约束

4

能跑，但难维护

短期可用，长期技术债务堆积

很多时候，问题不是 AI 完全不会做，而是它很容易做得"像对的一样"。
所以我们需要一套流程，不是为了拖慢 AI，而是为了把风险控制住。

EXECUTE

1

先审查计划

核心循环

Agent 是怎么一轮轮工作的？

// TS 伪代码
while (!taskComplete) {
  const response = LLM(messages, tools);

  if (response.toolCalls) {
    const results = executeTools(
      response.toolCalls
    );
    messages.push(...results);
  } else {
    return response.content;
  }
}

循环流程

1
读取当前上下文
2
调用 LLM 判断下一步
3
是否需要调用工具？
是 →
执行工具，结果放回上下文
否 →
返回结果

Agent 不是一次性把答案想完，而是带着最新上下文一轮轮继续判断和执行。

上下文累积

messages 为什么会越来越长？

第一次传给 LLM
- system:
你是一个 agent，需要优先使用工具获取实时信息。
- user:
帮我查一下杭州今天的天气。
LLM 返回
- tool call:
name: "getWeather"
arguments: { city: "杭州" }
第二次传给 LLM
- system: ...
- user: 帮我查一下杭州今天的天气。
- assistant:
调用工具 getWeather({ city: "杭州" })
- tool:
杭州，晴，25°C，东北风 2 级。

下一轮不是重新开始，而是把上一轮的 tool call 和工具结果一起带回给 LLM。

11 / 18

工具系统

LLM 为什么不只是会回答，还能调用工具？

// 工具定义与调用
const tools = {
  getWeather: {
    description: "查询指定城市的实时天气",
    execute: ({ city }) => {
      return `${city}，晴，25°C`;
    }
  }
};

const call = {
  name: "getWeather",
  args: { city: "杭州" }
};
const result = tools[call.name]
  .execute(call.args);

可以把 tools 理解成一张菜单

  • tools菜单上有哪些服务可以点
  • name点的是哪一项服务
  • args这次具体要什么
  • execute()后台真正去完成这项服务
  • result最后返回给模型的结果

"LLM 更像是在下单，不是在亲自做事。"

LLM 负责决定"调用什么"，外层程序负责真正"把事情做完"。

一句话总结

Agent 不是魔法，
它只是把"会回答"变成了"会行动"。

1

每一轮都带着最新上下文重新调用 LLM

2

LLM 决定是否调用工具

3

工具结果再回到上下文里进入下一轮
