## 开场（约 3 分钟）

大家好，今天想跟大家聊聊 AI 在前端开发流程中的应用。

先说一个我的观点：**AI 不会取代开发者，但会使用 AI 的开发者会取代不会使用的。**

今天的内容不是给大家推销某个工具，而是分享我实际工作中如何将 AI 融入开发流程的。会围绕三个部分展开：

1. **我的 AI 辅助开发工作流** —— 我怎么用 AI 写代码
2. **AI Agent 的运行原理** —— AI 写代码背后的"黑盒"是什么
3. **我的 AI 学习方法** —— 如何跟上这个快速变化的领域

## 第一部分：我的 AI 辅助开发工作流（约 15 分钟）

### 1.1 核心理念：人负责想清楚，AI 负责动手写

我先把结论放出来：

```
┌─────────────────────────────────────────────────┐
│                                                   │
│   人的职责：想清楚做什么、验证做得对不对           │
│   AI 的职责：动手写代码、执行重复劳动              │
│                                                   │
│   人 = 架构师 + 审查员                             │
│   AI = 执行者 + 助手                               │
│                                                   │
└─────────────────────────────────────────────────┘
```

这不是偷懒，而是重新分配精力。以前我 80% 的时间在写代码、20% 在思考；现在我 80% 的时间在思考和验证，20% 在写代码。

### 1.2 需求阶段：让 AI 先做分析，再动手

拿到一个新需求的时候，我不会让 AI 直接写代码。我的流程是这样的：

**第一步：告诉 AI 这次的需求是什么**

我会先自己读一遍 PRD 文档（PDF 格式的产品需求文档），总结出核心要点，然后告诉 AI：

> "这次的需求是做一个用户登录页面，支持手机号验证码登录和微信授权登录，需要考虑异常状态处理。"

这一步很关键 —— 你得先理解需求，才能给 AI 正确的指令。

**第二步：告诉 AI 约束条件**

> "我们的技术栈是 React + TypeScript，UI 组件库用的是 Ant Design，状态管理用 zustand，不能用 Redux。路由用 React Router v6。样式方案用 CSS Modules。"

这些约束条件确保 AI 产出的代码能融入我们现有项目，而不是生成一堆没法用的"demo 代码"。

**第三步：把完整的 PRD 文档发给 AI**

**第四步，也是最重要的一步：告诉 AI —— "先不要写代码"**

> "请分析这个需求，输出以下内容：
> 1. 需要实现哪些功能模块？
> 2. 需要新增或修改哪些文件？
> 3. 会影响现有哪些功能？
> 4. 有哪些潜在风险或需要确认的点？"

**为什么不让 AI 直接写代码？**

因为直接写出来的代码，你不知道它是不是理解对了需求。让它先分析，你先审查分析结果，确认方向对了再让它动手。这样能避免大量返工。

打个比方：这就像装修房子，你会先让设计师出图纸确认，而不是直接让工人开始砌墙。

### 1.3 开发阶段：分步执行，逐步验证

分析确认后，我不会让 AI 一次性把所有代码写完。而是拆分成小步骤：

```
需求分析确认 → 写类型定义 → 写组件骨架 → 写业务逻辑 → 写样式 → 联调
     ↑              ↑             ↑            ↑          ↑        ↑
   人工确认      人工确认      人工确认      人工确认    人工确认   人工确认
```

每一步我都会：
1. 看 AI 写的代码
2. 理解它的逻辑
3. 确认没有问题
4. 再进行下一步

这个过程看起来"慢"，但实际上比直接写快很多。因为：
- AI 写一个组件可能 30 秒，我自己写可能 30 分钟
- 审查 AI 的代码比从头写快得多
- 分步执行可以把错误限制在小范围内

### 1.5 这个工作流对不同角色的启发

我分享的是前端开发的流程，但这个思路对大家都有参考价值：

| 角色 | AI 可以做什么 | 人需要做什么 |
|------|--------------|-------------|
| **产品** | 辅助写 PRD、梳理需求、竞品分析 | 定义产品方向、做决策、验证 AI 输出 |
| **UI 设计** | 生成设计稿参考、配色建议、图标生成 | 审美把控、交互设计、品牌一致性 |
| **前端开发** | 写代码、调试、写测试 | 架构设计、需求理解、代码审查 |
| **后端开发** | 写接口、写 SQL、写测试 | 系统设计、性能优化、安全审查 |
| **测试** | 生成测试用例、自动化脚本 | 测试策略、边界场景、质量把关 |

核心思路是一样的：**人是大脑，AI 是手脚。**

## 第二部分：AI Agent 的运行原理（约 15 分钟）

### 2.1 从聊天机器人到 Agent

上面我说的那些工作，是用一个叫"Agent"的东西完成的。那 Agent 和我们平时用的 ChatGPT 聊天有什么区别？

**普通聊天机器人：只能回复文字**

```
你: 帮我看看这个文件
机器人: 这个文件的内容是 xxx（但如果你不给它文件，它就看不到了）
```

**Agent：可以自主完成任务，调用工具**

```
你: 帮我看看项目里有哪些组件
Agent: [思考] 我需要查看文件系统...
Agent: [行动] 读取项目目录
Agent: [观察] 发现了 15 个组件文件
Agent: [完成] 项目中有 15 个组件，分别是 xxx
```

区别在哪？Agent 能自己决定要做什么、调用什么工具，然后根据结果继续思考。

### 2.2 Agent 的核心公式

一句话理解 Agent：

```
Agent = 大模型 + 工具 + 循环
        (思考)   (行动)  (观察)
```

用我实际的工作来举例：

```
我: "把这个组件的按钮颜色改成蓝色"

Agent 第1轮:
  [思考] 用户要改按钮颜色，我需要先找到这个组件的文件
  [行动] 搜索文件系统，找到 Button.tsx
  [观察] 找到了文件: src/components/Button.tsx

Agent 第2轮:
  [思考] 我需要读取这个文件，看看当前的样式定义
  [行动] 读取 Button.tsx 的内容
  [观察] 文件内容中按钮颜色是 color: red

Agent 第3轮:
  [思考] 需要把 color: red 改成 color: blue
  [行动] 修改文件
  [观察] 修改成功

Agent 第4轮:
  [思考] 任务完成了，可以告诉用户了
  [完成] 已经把按钮颜色从红色改成了蓝色
```

### 2.3 ReAct 原理：边想边做

这个"思考 → 行动 → 观察"的循环，有个学术名字叫 **ReAct**，来自一篇 2022 年的论文。

**ReAct = Reasoning（推理）+ Acting（行动）**

| 阶段 | 含义 | 例子 |
|------|------|------|
| **Reasoning** | 分析问题，决定下一步 | "用户要改代码，我需要先找到文件" |
| **Acting** | 调用工具，执行操作 | "读取文件内容" |
| **Observing** | 获取结果，继续思考 | "看到了文件内容，颜色在第 15 行" |

这其实和我们人类解决问题的思路是一样的。比如你修一个 bug：

1. 先想：这个 bug 可能是什么原因？（思考）
2. 打开浏览器控制台看报错信息（行动）
3. 看到报错是"undefined is not a function"（观察）
4. 再想：哪里调用了 undefined？（思考）
5. 打开代码文件看（行动）
6. 找到了问题所在（观察）
7. 修复它（行动）

### 2.4 Agent 能用哪些工具？

不同 AI 编程工具提供的工具不一样，但核心的差不多：

| 工具 | 作用 | 类比 |
|------|------|------|
| **文件读取** | 读取项目中的文件 | 你用编辑器打开文件 |
| **文件写入** | 创建或修改文件 | 你保存文件 |
| **代码搜索** | 在项目中搜索关键词 | 你用 Ctrl+F 或 grep |
| **终端执行** | 运行命令行命令 | 你在终端里输入命令 |
| **网页搜索** | 搜索互联网获取信息 | 你 Google 搜东西 |

Agent 就是把这些工具组合起来用，像人一样操作电脑完成开发任务。

### 2.5 为什么要了解这个原理？

大家可能会想，我又不开发 AI 工具，为什么要知道这些？

因为理解原理能帮你：

1. **更好地使用工具** —— 知道 Agent 是分步思考的，你就知道应该给它清晰的、分步的指令，而不是一句话让它做十件事
2. **判断 AI 的输出** —— 知道 AI 是基于已有信息推理的，你就知道它可能"看漏了"某些文件，需要你提醒
3. **设计工作流** —— 知道 Agent 有哪些工具，你就能设计出更高效的协作流程

简单说：**你越了解它，就越能驾驭它。**

## 第三部分：我的 AI 学习方法（约 10 分钟）

### 3.1 面对快速变化的领域，怎么学？

AI 领域变化太快了，几乎每周都有新工具、新模型、新论文。如果试图"跟上所有东西"，你会被淹没。

我的策略很简单：

```
关注源头 → 动手实践 → 深度学习
   ↑            ↑           ↑
 GitHub      实际项目     源代码阅读
 Trending    中使用       + AI 辅助理解
```

### 3.2 关注源头：GitHub Trending

我有一个习惯，定期访问 GitHub Trending（github.com/trending），看看最近最火的项目有哪些。

这个页面的价值在于：
- 它是全球开发者的"风向标"
- 热门项目往往代表了最新的技术趋势
- 能帮你发现好用的工具和框架

比如最近很火的项目类型：
- AI Agent 框架（如 LangChain、AutoGPT）
- AI 编程工具（如 Cursor、Claude Code）
- 本地大模型部署工具（如 Ollama）

### 3.3 深度学习：clone 下来，AI 辅助学源码

发现感兴趣的项目后，我不会只是看看 README。我会：

1. **clone 项目到本地**
2. **用 AI 帮我读源码**

具体怎么操作？我会告诉 AI：

> "我正在学习这个项目的源码，请帮我分析：
> 1. 这个项目的整体架构是什么？
> 2. 核心模块有哪些，各自的职责是什么？
> 3. 数据流向是怎样的？
> 4. 有哪些值得学习的设计模式？"

AI 会像一个"耐心的导师"一样，一步步带你理解整个项目。

### 3.4 学习思路总结

```
第一步：发现
  → GitHub Trending / 技术社区 / 推特
  → 关注"最近有什么新东西"

第二步：评估
  → 看 README 和 Star 数
  → 判断"这个东西和我有关吗"

第三步：实践
  → clone 下来，跑起来
  → 自己先用一用

第四步：深入
  → 用 AI 辅助阅读源码
  → 理解"它是怎么实现的"

第五步：应用
  → 把学到的东西用在实际工作中
  → 写个小 demo 验证理解
```

### 3.5 给不同角色的建议

| 角色 | 推荐关注方向 |
|------|------------|
| **产品** | AI 产品案例、AI 辅助写 PRD 的方法 |
| **UI 设计** | AI 绘图工具（Midjourney、Stable Diffusion）、设计系统自动化 |
| **前端开发** | AI 编程工具（Cursor、Claude Code、GitHub Copilot） |
| **后端开发** | AI Agent 框架、LLM API 集成、RAG 系统 |
| **测试** | AI 生成测试用例、自动化测试框架 |

## 总结（约 3 分钟）

回顾一下今天分享的三个核心内容：

### 1. 工作流：人想清楚，AI 动手

```
需求 → 我理解总结 → 告诉 AI 约束 → AI 分析（不写代码）→ 我确认 → AI 分步执行 → 我逐步审查
```

关键点：
- 不要让 AI 直接写代码，先让它分析
- 给 AI 清晰的约束条件
- 分步执行，逐步验证

### 2. Agent 原理：思考 → 行动 → 观察

```
Agent = 大模型(思考) + 工具(行动) + 循环(观察)
```

关键点：
- Agent 不是魔法，是"有工具的 LLM"
- 理解原理能帮你更好地使用工具
- ReAct 循环和人类解决问题的思路一样

### 3. 学习方法：关注源头 + 深度实践

```
GitHub Trending → clone 项目 → AI 辅助学源码 → 应用到工作
```

关键点：
- 不要试图跟上所有东西
- 选择性深度学习比广度浏览更有价值
- AI 是学习的好帮手

### 最后的想法

AI 不是来取代我们的，而是来改变我们工作方式的。就像当年从手动写 HTML 到用框架开发，从 FTP 部署到 CI/CD 一样，每一次工具的进化都让我们把精力放到更重要的事情上。

**我们真正需要关注的，永远是"想清楚要做什么"和"验证做得对不对"。**
