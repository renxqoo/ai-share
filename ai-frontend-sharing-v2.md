## 第一部分：我的 AI 辅助开发工作流

### 整体框架：PEV

我用的是一个很简单的框架，三个阶段反复循环：

```
Plan  →  Execute  →  Verify
（计划）   （执行）    （验证）
```

跟传统开发的区别就一个核心原则：**人做决策，AI 做执行**。

这个原则听起来简单，但实际操作中很容易模糊边界——什么时候该让 AI 自由发挥？什么时候必须亲自上手？PEV 框架就是用来回答这个问题的。

***

### PLAN 阶段：先想清楚，再动手

**第一步：我告诉 Agent 三样东西**

1. **需求描述**：我会先自己把 PRD 看完理解，然后用自己的话总结成一段文字告诉 Agent——这个需求做什么？最终目标是什么？
2. **产品文档（PRD）**：完整原文也发给 Agent 做参考，避免我的总结遗漏细节
3. **约束条件**：哪些能做，哪些不能做，哪些不能改

**关键操作：禁止 AI 写代码**

这一步我会明确告诉 Agent："先不要写代码，只输出需求分析和实现方案。"

Agent 输出的方案里需要回答几个问题：

- 要实现什么功能？
- 需要改哪些文件？
- 会影响哪些现有模块？
- 技术方案有没有风险点？

**第二步：我仔细看完方案，确认无误才放行**

这一步不能被省略。AI 可能会误读 PRD，可能会提出不符合项目架构的方案。人的判断在这里不可替代。

> 📋 **案例模板**（此处填入实际案例）：
>
> - **任务背景**：什么需求？涉及几个模块？
> - **AI 方案输出**：AI 提出了什么方案？有没有你想调整的地方？
> - **方案调整**：你改了哪里？为什么？
> - **小结**：如果让人从零想方案要多久？AI 帮你省了多少时间？

**第三步：方案沉淀为 Spec，持久化到项目里**

确认后的方案不要只留在对话记录里。我会把它写成一份 Spec 文档，提交到代码仓库。这样做有三个好处：

1. **跨会话复用**：你关掉终端再开，Agent 还能读到之前的方案，不用从头沟通
2. **团队协作**：其他同学接手你的任务时，能看到完整的设计意图和决策背景
3. **可追溯**：出了问题可以回溯"当初为什么这么设计"

具体做法：在项目根目录维护一个 `CLAUDE.md`（或 `AGENTS.md`）文件，里面记录：

- 项目的技术栈和架构约定
- 代码风格和目录结构规范
- 已确认的技术方案和关键决策

现在越来越多人在讲 **Spec-Driven Development（规格驱动开发）**——把需求规格当成和代码一样重要的工程制品，纳入版本控制。简单说：**spec 不是一次性输入，而是项目资产。**

***

### EXECUTE 阶段：给定约束，让 AI 写代码

方案确认后，开始让 AI 写代码。这时候给的约束条件会更具体：

- 遵循当前项目的代码风格
- 不能有 mock 数据
- 合理拆分代码，组件化开发（单文件不超过 300 行）
- 不要写兼容代码（不引入不必要的抽象）

约束给得越细，AI 输出的代码越能直接用，返工越少。

**代码写完，立刻写测试**

AI 写完代码后，紧接着让它写单元测试。约束条件：

- 每个功能写完都要写单元测试
- 每个测试用例覆盖所有边界情况
- 每个测试用例都必须通过

前端项目强校验防线：ESLint + TypeScript + Prettier + Vitest。代码不符合规范直接报错，AI 自动修复直到通过。

***

### VERIFY 阶段：人工把关，不能少

代码和测试都写完了，但这时候还不能算结束。真正决定这次交付质量的，往往就是 Verify 这一段。我会逐项检查：

**Code Review：**

- 是否符合现有代码风格和架构规范
- 是否引入了重复逻辑（AI 容易埋头写，不管已有实现）
- 有没有安全隐患（硬编码密钥、缺少输入校验）
- 类型定义是否严谨
- 有没有潜在的副作用

**功能自测：**

- 整个流程走一遍，流程是否完整
- 交互是否流畅，有没有卡顿
- loading 状态展示是否合理
- 错误提示是否用户友好
- 各种屏幕尺寸下布局是否正常

**如果自测里发现 Bug：**

我的做法不是直接让 AI 去“试着修一下”，而是先让它把 Bug 稳定复现出来，再进入修复。

- 先写一个失败的测试，把问题钉住
- 再让 AI 改代码，直到测试通过
- 最后再回到自测，确认这次是真的修好了

```
发现 Bug → 写一个失败的测试（复现）→ 让 AI 修代码 → 测试通过 → 回到 Verify 确认修复
```

这样做的好处是，修 Bug 不靠感觉，而是靠一个可以重复验证的闭环。

### 也要诚实面对：AI 不总是靠谱

说完了工作流，我想专门留一点时间聊聊 **AI 翻车的场景**。这不是泼冷水，而是想让大家对它的能力边界有一个更真实的预期。

**AI 常见的翻车模式：**

1. **"自信的错误"** — AI 会非常自信地给出一个看起来合理但实际有问题的方案。比如把一个本该用 `useMemo` 的计算放在了渲染循环里，表面上跑得通，但性能隐患很大。你不仔细看，根本发现不了。
2. **重复造轮子** — AI 不知道你项目里已经有一个现成的工具函数，它会重新写一个功能相同但实现不同的版本。结果项目里出现两份做同样事的代码。
3. **忽视上下文** — AI 可能会改了一个文件，但没意识到这个改动会影响另外三个文件。它的视野是有限的，跨模块的影响需要人来把控。
4. **"幻觉"问题** — 有时候 AI 会引用一个根本不存在的 API，或者使用一个已废弃的方法。它自己并不知道自己"不知道"。

**所以 PEV 工作流里，Verify 阶段为什么这么重？就是因为这些翻车场景是真实存在的。** 人不是在 PEV 里做"橡皮图章"式的审批，而是做真正的质量把关。

> 📋 **翻车案例模板**（此处填入实际案例）：
>
> - **场景**：什么需求？
> - **AI 做了什么**：看起来对的方案
> - **实际问题**：你是怎么发现的？
> - **教训**：如果没发现会怎样？

***

以上就是我自己现在比较稳定的一套 AI 辅助开发工作流。聊完“怎么用”，下面我们把视角再往里收一点，看看它背后到底是怎么工作的，这样很多现象就更容易理解了。

## 第二部分：AI Agent 的运行原理

> 这一部分面向所有同学，不用技术背景也能听懂。

### 先看一个日常场景

比如你在终端里跟一个 AI 编码助手说："帮我在用户列表页加一个搜索框。"

然后你看到它自己去读文件、分析代码、写改动、写测试、跑检查……最后告诉你做好了。

你有没有想过：**它到底是怎么做到这一串操作的？**

### 一句话理解 Agent

```
Agent = 大模型 + 工具 + 循环
```

普通聊天机器人只能回复文字。Agent 不同——它**能做事**。

举个例子：

```
普通 LLM：
  你: "帮我查北京天气"
  它: "北京今天晴，25度。"（这是训练数据里的知识，可能是过期的）

Agent：
  你: "帮我查北京天气"
  Agent：[思考] 我需要查询天气...
  Agent：[行动] 调天气 API 查
  Agent：[观察] API 返回：晴，25度
  Agent：[回答] 北京今天晴，25度。
```

区别在哪？普通 LLM 靠记忆回答，Agent 靠工具行动。记忆可能过期，工具获取的是实时数据。

### 核心：ReAct 原理

Agent 的工作方式叫 **ReAct = Reasoning（思考）+ Acting（行动）**

就是一个"边想边做"的循环：

| 阶段     | 做什么         | 对应到实际表现          |
| ------ | ----------- | ---------------- |
| **思考** | 分析现状，决定下一步  | 判断“我应该先看哪里、先做什么” |
| **行动** | 调用工具执行操作    | 读文件、改文件、运行命令     |
| **观察** | 拿到结果，评估是否继续 | 看结果对不对，再决定下一步    |

**用加搜索框的例子走一遍：**

```
第 1 轮：
  思考：需要知道用户列表页长什么样
  行动：先去读相关页面文件
  观察：读到了 200 行代码，里面有表格组件

第 2 轮：
  思考：要用 Ant Design 的 Input.Search 组件，在表格上方加搜索栏
  行动：修改页面代码
  观察：代码改完了

第 3 轮：
  思考：需要写测试
  行动：补测试代码
  观察：测试写完了，跑一下

第 4 轮：
  思考：lint 报了一个类型错误
  行动：继续修代码
  观察：lint 通过了

第 5 轮：
  思考：功能已实现，测试通过，lint 通过，任务完成
  行动：无 → 直接回答"已完成"
```

这就解释了为什么这类 AI 编码助手能自己去读文件、改文件、修问题——它在重复 "思考 → 行动 → 观察" 这个循环，直到任务完成。

### 三个关键技术点

**1. 工具（Tools）**

Agent 能用的工具不止是“输出一段文字”，它通常还能直接调用一些执行能力，比如：

- 读文件
- 改文件
- 执行命令（跑测试、装依赖、做代码检查）
- 搜索代码
- 按文件名查找内容

工具本身会告诉 AI“什么时候该用、怎么用”。

**2. 消息历史（记忆）**

Agent 维护一个完整的消息记录。每一轮思考它都能看到前面的所有对话和操作结果。这就是为什么它能做多步操作不会"忘"。

**3. 循环上限**

Agent 不是无限循环的。这类工具通常都会设置一个上限，防止它一直绕下去。如果超了还没完成，它就会停下来让你介入。

### 这一部分小结

你不需要记住所有细节，但可以带走两个认知：

1. **AI 写代码不是魔法**，是"思考→行动→观察"的循环，你在流程图的每一轮都能看到它的决策
2. **人在环路上的位置**：你制定规则（约束条件），AI 在规则内循环执行，最终你验证结果

## 第三部分：我的 AI 学习方法

> 这一部分很短，因为我的方法确实不复杂。

### 核心方法：GitHub Trending + AI 辅助读源码

**第一步：定期刷 GitHub Trending**

每周花十几分钟扫一眼 [github.com/trending](https://github.com/trending)，看最近最火的项目有哪些。不是为了追热点，而是保持对"大家都在关注什么"的感知。

关注点：

- 这个项目解决什么问题？
- 有没有新的开发范式或工具思路？
- 跟我现在的技术栈有没有关系？

**第二步：对感兴趣的项目，clone 下来 + AI 辅助读源码**

传统方式：clone 下来，从 README 开始啃，看目录结构，一点一点读源码，半天过去了还没搞清楚核心逻辑。

我的方式：

```
1. git clone <项目地址>
2. 把项目目录喂给 AI Agent
3. 问 AI：
   - "这个项目的核心架构是什么？"
   - "最关键的代码在哪个文件？"
   - "XX 功能是怎么实现的？"
   - "这个设计为什么这么写？有什么好处？"
4. AI 帮你做代码导航 + 讲解，你跟着它的分析读源码
```

> 📋 **案例**：我用这个方法读过一些 Agent 和 CLI 工具的源码。传统方式可能需要好几天，用 AI 辅助，几个小时就能先把核心结构、关键模块和设计思路摸清楚。

**为什么这个方法有效？**

- 读源码是最直接的学习方式，但门槛高（看不懂、找不着入口）
- AI 帮你降低了"看不懂"的门槛，相当于有一个随时可以问的导师
- 你是主动探索，不是被动输入——理解深度不一样

### 第二个方法：关注几个高质量信息源

除了读源码，我还保持着几个固定的信息渠道：

- **GitHub Changelog / 技术博客**：关注你使用的核心工具的更新日志（React、TypeScript、各类 AI 编码工具等）。不用每篇都读，扫标题，感兴趣的才深入
- **技术社区的精选内容**：比如 Hacker News、掘金的热门文章。不是追热点，而是了解"社区在讨论什么"
- **官方文档优先**：遇到新工具新概念，**先看官方文档再看博客**。官方文档是最准确的，二手博客可能已经过时

**一个重要的心态：不需要跟上所有东西。**

这个领域每天都有新东西出来，追不完的。我的策略是：**保持感知，按需深入**。知道有这个东西存在就行，等真正需要用的时候再花时间学。不要有 FOMO（Fear of Missing Out），AI 工具的学习曲线实际上越来越低——今天的工具比半年前的工具好用得多，所以不用急着把每个新工具都试一遍。

***

以上就是我个人比较常用的学习方法。最后一部分，我不讲日常开发细节了，而是往团队层面再延伸一步，聊一个我觉得值得尝试的方向。

## 第四部分：团队级延伸实践

> 这一部分不是前端日常开发流程本身，而是沿着前面的思路，往团队协作和业务系统层面延伸。

这一部分我想抛出一个想法：把我们的 SaaS 系统的业务能力暴露给 Agent，让它直接操作系统。

### 思路：把业务操作变成 CLI 命令

我们现在的系统，所有操作都只能通过网页界面完成——查付款单、看交易记录、审批流程、查看余额，都要登录系统点来点去。

但如果我们把常用的业务操作封装成 CLI 命令，比如：

```bash
saas payment list --status=pending          # 查看待处理的付款单
saas payment detail --id=PAY20260509001     # 查看某笔付款详情
saas deal quote --pair=USD/CNH              # 查询实时汇率
saas deal execute --pair=USD/CNH --amount=10000 --direction=buy  # 执行交易
saas approval list --pending                # 查看待审批列表
saas approval approve --id=APV001           # 审批通过
saas balance check --currency=USD           # 查询余额
saas beneficiary list --customer=CUST001    # 查询收款人列表
```

这些命令本质上就是调用我们现有的 API 接口。我们已经有完整的 `services/` 层，后端 API 都在，只需要包一层 CLI。

### 这跟 Agent 有什么关系？

一旦业务操作变成了 CLI 命令，Agent 就能**直接操作我们的业务系统**。你不需要打开浏览器、登录系统、点点点，而是在终端里用自然语言告诉 Agent：

```
你: "查一下今天有多少笔待审批的付款"
Agent: [执行 saas approval list --pending]
       当前有 12 笔待审批付款，其中 3 笔超过 24 小时未处理

你: "把金额最大的那笔详情给我看"
Agent: [执行 saas payment detail --id=PAY20260509087]
       付款编号: PAY20260509087
       金额: USD 500,000
       收款人: ABC Trading Co., Ltd.
       状态: 待审批

你: "这笔交易的汇率是多少？"
Agent: [执行 saas deal quote --pair=USD/CNH]
       当前 USD/CNH 汇率: 7.2450
```

更进一步，Agent 可以做**自动化流程**：

```
你: "帮我监控 USD/CNH 汇率，当汇率低于 7.20 时自动买入 10 万美元"
Agent: 收到，开始监控... [后台持续运行]
       14:32 汇率触及 7.198，已自动执行买入
       交易编号: DEAL20260509001，成交价 7.198，金额 USD 100,000
```

或者：

```
你: "每天早上 9 点把昨天的交易汇总发给我"
Agent: [定时执行 saas report daily --date=yesterday]
       昨日交易汇总：
       总交易笔数: 47
       总交易金额: USD 2,340,000
       待处理: 5 笔
       异常标记: 1 笔（大额交易待复核）
```

### 这个想法的实质

让 Agent 成为业务系统的操作者。人定规则（什么时候买、审批条件是什么、什么情况需要通知），Agent 在规则内自动执行。

我们项目里已经有现成的 API 接口层（`services/`），CLI 只是加一层薄壳。这件事技术上不难，难的是想清楚：

- **哪些操作适合暴露给 Agent** — 查询类操作风险低，适合先做；交易类操作需要权限控制
- **权限怎么控制** — 不同角色能执行不同命令，需要跟现有的权限体系打通
- **审计日志** — Agent 的每次操作都要有记录，出了问题可以追溯

这些恰恰是值得团队讨论的。

### 也可以换个角度想

刚才说的是后端 API → CLI → Agent。对于在座的不同角色，其实可以有不同的切入思路：

- **前端同学**：用 Agent 做组件开发、页面还原、自动化测试——这是我今天讲的工作流
- **后端同学**：把 API 封装成 CLI，让 Agent 能直接操作业务——这是刚才说的思路
- **产品同学**：用 AI 做需求分析、竞品调研、PRD 草拟——AI 在文字工作上已经很成熟了
- **测试同学**：用 AI 生成测试用例、分析测试覆盖率、自动化回归测试
- **设计同学**：用 AI 做设计稿到代码的转换、设计灵感探索

**核心思路是一样的：把 AI 当成一个执行力很强的助手，你定规则，它做执行。**

> 📋 **建议下一步**：可以先挑一个最简单的场景做原型——比如把 payment list 做成 CLI 命令，让 Agent 能通过终端查询付款列表。验证通了这个思路，再扩展到更多业务操作。

## 结束语

今天我主要想带大家看四件事：

1. **PEV 工作流**：Plan → Execute → Verify，人做决策，AI 做执行
2. **Agent 原理**：思考 → 行动 → 观察，AI 写代码不是魔法，是可理解的循环
3. **学习方法**：GitHub Trending 保持感知，AI 辅助读源码深入理解
4. **团队实践**：把 SaaS 业务操作封装成 CLI，让 Agent 直接操作系统

如果今天只带走一句话，我希望是这句：**AI 能放大你的能力，但不能替代你的判断。**

工具会越来越强，速度也会越来越快，但真正决定结果质量的，还是你怎么拆问题、怎么定规则、怎么做判断、怎么做最后那道把关。

所以对我们来说，重点不是“要不要用 AI”，而是“怎么把它放进一个可控、可复用、可协作的流程里”。
