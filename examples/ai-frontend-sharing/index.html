PART 01

我的 AI 辅助开发工作流

一个核心原则：人做决策，AI 做执行

好，进入第一部分。我先介绍一个我用了很久的框架，叫 PEV——Plan、Execute、Verify，计划、执行、验证。

这个框架的核心原则就一句话：人做决策，AI 做执行。听起来简单，但实际操作中很容易模糊边界——什么时候让 AI 自由发挥？什么时候必须亲自上手？PEV 就是用来回答这个问题的。

// framework

PEV：Plan → Execute → Verify

Plan · 计划

准备上下文 → AI 出方案 → 人工 Review 方案

禁止 AI 写代码，只输出需求分析和实现方案

Execute · 执行

AI 写代码 → AI 写测试 → 自动修复循环

约束给得越细，返工越少

Verify · 验证

Code Review + 功能自测 + 体验检查

人工把关，不能省

PEV 分三个阶段。Plan 阶段，我告诉 AI 需求是什么、约束是什么，让它输出方案。这里有个关键操作——禁止 AI 写代码，只让它出方案。

为什么要禁止？因为你得先确认方案是对的，再让它动手。方案不对，代码写得再快也是白费。

Execute 阶段，方案确认后，才让 AI 写代码、写测试。这时候给的约束会更具体——代码风格、单文件不超过 300 行、不能有 mock 数据。约束越细，输出越能直接用。

Verify 阶段，代码写完了，但我的工作没完。我会逐项做 Code Review、功能自测、体验检查。这个阶段是人的主场，AI 替代不了。

// plan phase

PLAN：先想清楚，再动手

①
告诉 Agent 三样东西

需求描述（自己总结）+ PRD 原文 + 约束条件

②
禁止 AI 写代码

只输出需求分析和实现方案，回答四个问题：做什么？改哪些文件？影响谁？有风险吗？

③
人工确认方案

AI 可能误读 PRD、提出不符合架构的方案。人的判断不可替代。

Spec-Driven Development

确认后的方案写成 Spec 文档，提交到代码仓库。

好处：① 跨会话复用 ② 团队协作 ③ 可追溯

具体做法：项目根目录维护 CLAUDE.md

Plan 阶段我怎么做？第一步，告诉 Agent 三样东西：我自己理解的需求描述、PRD 原文、约束条件。注意需求描述一定要自己先消化一遍再告诉 AI，不要直接扔 PRD 过去。

第二步，明确告诉 AI：先不要写代码，只输出方案。方案里要回答四个问题——做什么、改哪些文件、影响哪些模块、有没有风险。

第三步，我仔细看完方案，确认无误才放行。这一步不能省。AI 真的会误读 PRD，真的会提不靠谱的方案。

右边这个卡片讲的是一个重要的实践——Spec-Driven Development。确认后的方案不要只留在对话里，要写成文档提交到仓库。这样你下次开终端 AI 还能读到，其他同学接手也看得见。

// execute phase

EXECUTE：给定约束，让 AI 写代码

约束条件越细，输出越可用

遵循项目代码风格
不能有 mock 数据
单文件不超过 300 行
不引入不必要的抽象

写完代码，立刻写测试

每个功能都要有单元测试，每个测试都要覆盖边界情况，每个测试都必须通过。

强校验防线：ESLint + TypeScript + Prettier + Vitest

不符合规范直接报错，AI 自动修复直到通过。

方案确认后，进入执行阶段。这时候给 AI 的约束会更具体。左边列了几个典型的约束——代码风格、不能有 mock 数据、单文件不超过 300 行、不要写兼容代码。

约束给得越细，AI 输出的代码越能直接用，返工越少。这个经验非常重要。你如果不给约束，AI 就按它自己的习惯写，可能跟项目风格完全不搭。

右边强调一个点：代码写完，立刻让 AI 写测试。不要等，不要想着"后面再补"。每个功能写完马上写测试，覆盖边界情况，跑通为止。

前端项目有 ESLint + TypeScript + Prettier + Vitest 这套强校验防线，代码不符合规范直接报错，AI 会自动修复直到通过。这个循环是完全自动的，不需要你干预。

// verify phase

VERIFY：人工把关，不能少

Code Review

✓ 代码风格和架构规范
✓ 是否引入重复逻辑（AI 容易不管已有实现）
✓ 安全隐患（硬编码密钥、缺少输入校验）
✓ 类型定义是否严谨
✓ 潜在的副作用

功能自测

✓ 整个流程走一遍
✓ 交互是否流畅，有没有卡顿
✓ loading 状态展示是否合理
✓ 错误提示是否用户友好
✓ 各屏幕尺寸下布局是否正常

执行完之后，进入 Verify 阶段。这是人最核心的工作。

左边是 Code Review 的检查清单。我特别标了一个——是否引入重复逻辑。这是 AI 最容易犯的错：它不知道你项目里已经有这个功能了，会重新写一份。所以 Review 的时候一定要看有没有"重复造轮子"。

右边是功能自测。不要只测"功能能不能用"，还要看交互体验——loading 状态有没有、错误提示用户看不看得懂、不同屏幕尺寸下会不会崩。

这些是 AI 做不了的。AI 能跑测试、能过 lint，但它不知道"这个 loading 转 5 秒用户会烦"。所以 Verify 阶段，人的判断不可替代。

// verify · hidden risks

隐性风险：三重债务模型

Martin Fowler & Storey, 2026 — AI 辅助开发带来的隐性成本

技术债务

AI 生成的 PR 比人类写的多 1.7 倍重大问题——控制流错误、安全漏洞。Code Review 可以覆盖。

认知债务

代码不是你写的，你对它理解不够深。久而久之团队不敢改代码——系统变成黑盒。

意图债务

代码能表达"做了什么"，但表达不了"为什么这么做"。设计意图在流失。

Verify 阶段除了检查常规问题，还要注意一些隐性风险。2026 年 Martin Fowler 和 Storey 提出了一个模型叫三重债务模型，专门描述 AI 辅助开发带来的隐性成本。

第一种是技术债务，这个最好理解——AI 写的代码能跑但不够干净。行业数据显示 AI 生成的 PR 比人写的多 1.7 倍重大问题。这个我们在 Code Review 里已经在查了。

第二种是认知债务，这个更隐蔽。代码不是你一行行写的，你对它的理解就不够深。久而久之团队会变得不敢改代码——怕改了这里，那里出问题。系统变成了一个黑盒。

第三种是意图债务——代码能表达"做了什么"，但表达不了"为什么这么做"。所以 Spec 持久化特别重要，它在保留设计意图。

// honest talk

也要诚实面对：AI 不总是靠谱

①
自信的错误

非常自信地给出一个看起来合理但实际有问题的方案。比如把该用 useMemo 的计算放在渲染循环里。

②
重复造轮子

不知道项目里已有工具函数，重新写一个功能相同但实现不同的版本。

③
忽视上下文

改了一个文件，没意识到这个改动会影响另外三个文件。跨模块影响需要人把控。

④
"幻觉"问题

引用不存在的 API，或使用已废弃的方法。它自己不知道自己"不知道"。

说完了工作流，我想特别聊一下 AI 翻车的场景。不是泼冷水，而是让大家对 AI 的能力边界有一个真实的预期。

第一种翻车叫"自信的错误"——AI 会非常自信地给出一个看起来完美但实际有隐患的方案。比如把一个计算密集的操作放在渲染循环里，表面上跑得通，但性能问题很大。你不仔细看根本发现不了。

第二种是重复造轮子——AI 不知道你项目里已经有一个做同样事的工具函数，它会重新写一份。结果是项目里出现两份重复代码。

第三种是忽视上下文——AI 改了一个文件，但没意识到这个改动会影响另外三个文件。

第四种是"幻觉"——AI 有时候会引用一个根本不存在的 API，或者使用已废弃的方法。

所以 PEV 里 Verify 阶段为什么这么重？就是因为这些翻车场景是真实存在的。

// fix · TDD for bugs

FIX：TDD 驱动修 Bug

①
先写一个失败的测试（复现 Bug）

用测试精确定位问题，不是凭感觉猜。

②
让 AI 修代码

AI 看到失败测试就知道该修哪里。

③
测试通过 = 修复确认

跑测试，过了就是过了。不用再猜"修好了没"。

发现 Bug → 写一个失败的测试（复现） → 让 AI 修代码 → 测试通过 → 确认修复 ✓

最后说一下修 Bug 的方法。如果自测发现 Bug，我的做法是：先让 AI 写一个能复现这个 Bug 的测试，再修代码。

为什么要先写测试？三个好处。第一，用测试精确定位问题，而不是凭感觉猜"应该是这里的问题"。第二，AI 看到失败的测试就知道该修哪里，不用你再解释半天。第三，改完之后跑一下测试就知道了，过了就是过了，不用再猜"到底修好没"。

这就是 TDD 的核心思路——先有测试，再改代码——只不过用在了 debug 场景。整个 PEV 工作流里，人始终在环路里做决策，AI 帮你跑得快。

PART 02

AI Agent 的运行原理

面向所有同学，不用技术背景也能听懂

好，以上就是我的 AI 辅助开发工作流。了解了怎么用 AI，你可能会好奇——它背后到底是怎么工作的？这就进入第二部分。

这一部分我保证不用任何高深的技术术语，不管你是什么岗位，都能听懂。我们从一个日常场景开始。

// agent 101

一句话理解：Agent = 大模型 + 工具 + 循环

大模型（思考）

理解你的意图，分析当前状况，决定下一步做什么。

工具（行动）

读文件、改文件、跑命令、搜索代码。不只是打字。

循环（观察）

拿到结果后继续思考，直到任务完成或需要你介入。

普通聊天机器人只能回复文字。Agent 不同——它能做事。

你在终端跟 Claude Code 说"帮我加个搜索框"，然后看到它自己去读文件、改文件、写测试、跑 lint……你有没有想过，它到底是怎么做到这一串操作的？

答案就三个词：大模型 + 工具 + 循环。大模型负责思考——理解你的意图，分析当前状况。工具负责行动——读文件、改文件、跑命令。循环负责观察——拿到结果后继续思考，直到任务完成。

普通聊天机器人只能回复文字。Agent 不一样——它能做事。这个区别很关键。你问普通 LLM"北京天气怎么样"，它靠训练数据回答，可能是过期的。你问 Agent，它会调天气 API 去查，拿到的是实时数据。

// react · reason + act

ReAct：思考 → 行动 → 观察

思考
Reasoning

分析现状，决定下一步做什么。比如"需要先看文件结构"。

行动
Acting

选择合适的工具执行操作。比如 read_file、edit、bash。

观察
Observation

拿到结果，判断还需要继续吗？任务完成了吗？

加搜索框的例子

R1 思考：用户列表页长什么样 → read_file
R2 思考：加 Input.Search 组件 → edit
R3 思考：写测试 → edit
R4 思考：lint 报类型错误 → edit
R5 思考：全部通过 → 完成

Agent 的工作方式有个名字叫 ReAct——Reasoning 加 Acting，思考加行动。就是一个"边想边做"的循环。

每一步都是三拍：思考当前状况，行动选工具干活，观察结果判断要不要继续。

右边是一个具体例子——让 AI 加一个搜索框。第一轮，它想"用户列表页长什么样"，于是去读文件。第二轮，想清楚了，开始改代码。第三轮，写测试。第四轮，发现 lint 报错，自动修复。第五轮，全部通过，告诉你做完了。

所以你看，AI 写代码不是魔法，就是这个循环在反复转。你在每一步都能看到它在想什么、做了什么。这就是为什么我们说"人在环路上"——你随时可以介入。

// under the hood

三个关键技术点

工具箱（Tools）

read_file 读文件
edit 改文件
bash 跑命令
grep 搜索代码
glob 按模式查找

每个工具都有描述，告诉 AI "什么时候用、怎么用"。

消息历史（记忆）

Agent 维护完整的消息记录。每一轮思考都能看到前面的所有对话和操作结果。

这就是为什么它能做多步操作不会"忘"。

循环上限

设置了最大迭代次数，防止"死循环"。

如果超了还没完成，会停下来让你介入。

这一页讲三个技术细节，不深入，有兴趣可以会后聊。

第一个是工具箱。Agent 不只是打字，它有具体的工具——读文件、改文件、跑命令、搜索代码。每个工具都有描述，告诉 AI 什么时候用、怎么用。这就像你给新同事一个工具清单，告诉他"改文件用 edit，找代码用 grep"。

第二个是消息历史。Agent 维护一个完整的记录，每一轮思考它都能看到前面发生了什么。这就是为什么它能做多步操作不会忘——你让它加搜索框，它改了文件还记得要去写测试。

第三个是循环上限。Agent 不是无限循环的，Claude Code 设了最大迭代次数。如果跑了很多轮还没完成，它会停下来让你介入。这是一个安全机制。

PART 03

我的 AI 学习方法

保持感知，按需深入

了解了 AI 的原理，下一个问题自然就来了——这个领域变化这么快，怎么跟得上？这就是第三部分。

这部分不长，因为我的方法确实不复杂。

// learning

GitHub Trending + AI 辅助读源码

第一步：定期刷 GitHub Trending

每周花十几分钟扫一眼。不是追热点，是保持对"大家都在关注什么"的感知。

这个项目解决什么问题？
有没有新的范式或工具思路？
跟我现在的技术栈有关系吗？

第二步：AI 辅助读源码

clone 下来 → 喂给 AI → 问架构、问核心文件、问实现细节。传统方式要好几天，AI 辅助几个小时搞定。

信息源管理

①
官方文档优先

遇到新工具新概念，先看官方文档再看博客。官方最准确。

②
技术社区精选

Hacker News、掘金热门。了解"社区在讨论什么"。

③
不需要跟上所有东西

保持感知，按需深入。不要 FOMO。

我的学习方法核心就两个。

第一个：定期刷 GitHub Trending。每周花十几分钟，看看最近最火的项目在解决什么问题。不是为了追热点，而是保持感知——知道大家在关注什么。遇到感兴趣的，clone 下来用 AI 辅助读源码。传统方式读一个项目要好几天，AI 辅助几个小时就能理解核心架构。

第二个：信息源管理。我保持几个固定渠道——官方文档优先于博客，技术社区看精选，GitHub Changelog 关注核心工具更新。

最后强调一个心态：不需要跟上所有东西。这个领域每天都有新东西，追不完的。我的策略是"保持感知，按需深入"。知道有这个东西就行，等真正需要用的时候再花时间学。AI 工具的学习曲线越来越低，不用急着每个都试。

PART 04

团队可以尝试什么

把 SaaS 业务能力暴露给 Agent

好，接下来聊一个更有意思的话题——我们团队可以怎么用 AI 做一些不一样的事。

这不是 PPT 上的概念，而是我们可以实际去做的。

// team practice

把业务操作变成 CLI 命令

saas payment list --status=pending # 查看待处理的付款单 saas payment detail --id=PAY001 # 查看付款详情 saas deal quote --pair=USD/CNH # 查询实时汇率 saas deal execute --pair=USD/CNH --amount=10000 # 执行交易 saas approval list --pending # 查看待审批 saas balance check --currency=USD # 查询余额

本质：调用现有 services/ 层 API，只加一层薄壳。

这个想法是这样的——我们现在所有操作都只能通过网页界面完成。但如果把常用的业务操作封装成 CLI 命令，事情就不一样了。

大家看屏幕上这些命令——查付款单、查汇率、执行交易、审批、查余额。这些命令本质上就是调用我们现有的 API 接口。我们已经有完整的 services 层，后端 API 都在，只需要包一层 CLI。

这件事技术上不难。难的是什么？下一页会讲。

// agent in action

需要想清楚的三个问题

① 哪些操作适合暴露？查询先做，交易需权限控制
② 权限怎么跟现有体系打通？
③ 每次操作都要有审计日志

一旦业务操作变成了 CLI 命令，Agent 就能直接操作我们的业务系统。看左边这个终端对话——你不用打开浏览器、不用登录系统，在终端里用自然语言告诉 Agent，它帮你查。

更进一步，Agent 可以做自动化流程——比如"监控汇率，低于某个值自动买入"，或者"每天早上把交易汇总发给我"。人定规则，Agent 自动执行。

但这件事有三个问题要想清楚。第一，哪些操作适合暴露给 Agent？查询类操作风险低，可以先做；交易类需要权限控制。第二，权限怎么跟现有体系打通？第三，审计日志——Agent 的每次操作都要有记录，出了问题可以追溯。

这些是值得团队讨论的。技术上不难，关键是想清楚业务规则。

最后收个尾。今天聊了四件事。

第一，PEV 工作流——Plan、Execute、Verify。人做决策，AI 做执行。这个框架你可以直接拿去用。

第二，Agent 原理——思考、行动、观察。AI 写代码不是魔法，就是这三步在反复循环。

第三，学习方法——GitHub Trending 保持感知，AI 辅助读源码深入理解。保持感知，按需深入。

第四，团队实践——把 SaaS 业务操作封装成 CLI，让 Agent 直接操作系统。技术上不难，关键是想清楚业务规则。

最后我想说：AI 工具进步速度比我们想象的快，但核心逻辑没变——它能放大你的能力，但不能替代你的判断。会写约束条件、能判断方案好坏、能发现隐患——这些才是你应该持续打磨的。
