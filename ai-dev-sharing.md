# AI 在前端项目中的实际应用——Vibe Coding 提效实践

## 一、开场（3 分钟）

### 我今天要讲什么

不是 AI 原理，不是未来趋势，而是：

> **我在日常开发中，怎么用 AI 把需求变成上线代码的**

一个前端工程师的真实工作流，从接到需求到交付上线，AI 参与了哪些环节，我是怎么和 AI 协作的。

### 为什么现在要聊这个

- AI 编程工具已经从"玩具"变成了"生产力工具"
- 但很多人还在停留在"让 AI 写个函数"的阶段
- 今天想分享的是：**怎么把 AI 用成一套完整的开发流程**

---

## 二、我之前的开发方式 vs 现在（5 分钟）

### 之前

```
看需求文档 → 自己理解需求 → 自己设计方案 → 手写代码 → 手写测试 → 自测 → 提测
```

- 大量时间花在"写代码"这个环节
- 方案设计全靠脑子里想，容易遗漏边界情况
- 单测写起来费劲，经常偷懒不写

### 现在

```
看需求文档 → 把需求+文档给 AI → AI 读代码出方案 → 我确认方案 → AI 写代码 → AI 写测试 → 我验证 → 提测
```

- 核心变化：**我从"写代码的人"变成了"审代码的人"**
- 代码量没少，但**我的时间从"写"转移到了"想"和"审"**
- 效率提升的核心不是 AI 多快，而是**我不用再从零开始**

### 一个直观的对比

| 环节 | 之前 | 现在（AI 辅助） |
|------|------|-----------------|
| 需求理解 | 自己读文档 | AI 帮我梳理要点 |
| 方案设计 | 自己想，可能遗漏 | AI 出方案，我审查补充 |
| 写代码 | 从零手写 | AI 生成初版，我调整 |
| 写单测 | 经常跳过 | AI 自动生成，我确认 |
| 调试排查 | 自己看日志 | AI 帮我定位问题 |

> **一句话总结**：AI 不是替代我，而是把每个环节的"起步成本"降到接近零。

## 三、我的完整工作流（15 分钟，核心环节）

这是今天分享的重点。我会用一个**真实需求**走一遍完整流程。

### 整体流程图

```
┌─────────────────────────────────────────────────────────────────┐
│                     AI 辅助开发完整流程                            │
│                                                                 │
│  1. 准备上下文 ──→ 2. AI 出方案 ──→ 3. 人工 Review 方案          │
│       ↑                                    │                    │
│       │                              确认无误                      │
│       │                                    ↓                    │
│  6. 最终验证 ←── 5. AI 写测试 ←── 4. AI 写代码                   │
│       │                                                         │
│       ↓                                                         │
│     提测上线                                                     │
└─────────────────────────────────────────────────────────────────┘
```

### 第一步：准备上下文，给 AI 充分的信息

这是很多人忽略的——**AI 的输出质量，取决于你给的输入质量**。

我会准备：

1. **需求描述**（用自然语言把需求说清楚）
2. **产品文档**（PRD 链接或关键截图）
3. **相关代码文件**（让 AI 知道现有代码结构）

**为什么这么做：**

> AI 不了解你的项目。你给它越多上下文，它出的方案越靠谱。
> 这一步花 5 分钟，后面省 50 分钟。

**实际操作示例（Prompt）：**

```
我需要实现一个 XXX 功能。

需求概述：
- 用户可以在 XXX 页面进行 XXX 操作
- 需要支持 XXX 和 XXX 两种场景
- 详细需求文档见附件

请先阅读以下代码文件，理解现有架构，然后给我一个修改方案。
不要写代码，只告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 有没有需要注意的边界情况
```

**关键点**：先让 AI 出方案，**不要直接让它写代码**。

---

### 第二步：AI 阅读代码，输出修改方案

AI 会返回类似这样的内容：

```
## 修改方案

### 1. 需要修改的文件
- src/pages/OrderList/index.tsx — 新增筛选条件
- src/services/order.ts — 新增接口调用
- src/types/order.d.ts — 新增类型定义

### 2. 修改思路
- OrderList 页面：在现有筛选栏基础上新增一个下拉选择器...
- order service：新增 fetchOrderFilter 接口...
- 类型定义：补充 FilterOption 类型...

### 3. 注意事项
- 现有的筛选逻辑用了 URL 参数同步，新增筛选条件也要保持一致
- 下拉数据需要做缓存，避免重复请求
- 移动端需要考虑下拉选择器的交互适配
```

---

### 第三步：人工 Review 方案（关键环节）

**这是整个流程里最重要的步骤。**

AI 的方案不是直接用的，我需要检查：

- 修改范围是否合理？有没有漏掉文件？
- 修改思路是否符合项目的架构规范？
- 它提到的注意事项有没有遗漏？
- 有没有更好的实现方式？

> **AI 出方案很快，但方案对不对，只有人能判断。**

如果方案没问题，进入下一步；如果有问题，我会补充信息让 AI 重新出方案。

---

### 第四步：AI 写代码

方案确认后，让 AI 开始写代码：

```
方案确认，请按照上面的方案开始写代码。
注意：
- 遵循项目现有的代码风格
- 使用项目中已有的工具函数，不要重复造轮子
- 类型定义要完整
```

AI 会逐个文件生成代码。这个阶段我主要关注：

- 代码风格是否和项目一致
- 有没有引入不必要的依赖
- 命名是否规范

---

### 第五步：AI 写测试

代码写完后，让 AI 补充测试：

```
请为新增的代码编写单元测试。
重点关注：
- 正常流程
- 边界情况（空数据、异常输入）
- 异常处理（接口失败、loading 状态）
```

> 之前手动写单测是最容易被跳过的环节，现在 AI 生成测试基本能覆盖 70-80% 的场景。

---

### 第六步：最终验证——人的兜底（最后一道防线）

**这是整个流程的最后一步，也是最容易被忽略的一步。**

我会做三件事：

#### 1. Code Review

不是简单"看一眼能不能跑"，而是检查：

- 是否符合现有代码风格和架构规范
- 是否引入了重复逻辑
- 是否有潜在副作用
- 类型定义是否严谨

> **标准不是"能不能跑"，而是"能不能长期维护"**

#### 2. 功能验证

| 验证类型 | 关注点 |
|---------|--------|
| 正常流程 | 核心功能是否完整可用 |
| 边界情况 | 空数据、极端输入、特殊字符 |
| 异常处理 | 接口失败、网络超时、loading/error 状态 |

> **测试用例是"理论覆盖"，手动验证是"真实体验"**

#### 3. 体验检查

这个是 AI 最容易忽略的部分：

- 交互是否流畅，有没有卡顿
- loading 状态展示是否合理
- 错误提示是否用户友好
- 各种屏幕尺寸下布局是否正常

---

### 这个流程的核心价值

```
AI 可以保证"生成"，但只有人能保证"正确 + 合理 + 可用"
```

```
没有最后这一步，前面所有提效都有可能变成"加速出错"
```

## 四、除了写需求，AI 还能做什么（5 分钟）

### 1. 修 Bug——先复现，再修复

这是一个我自己摸索出来的流程，和开发新需求不同，修 bug 有一个关键原则：

> **先让 bug 能被稳定复现，再动手改代码**

#### 以前的修 bug 方式

```
看报错信息 → 凭经验猜原因 → 改一版试试 → 不行再猜 → 反复几次 → 可能修好了，也可能引入新问题
```

问题在于：
- 不确定改的是不是真正的原因
- 改完之后没有验证"bug 确实被修复了"
- 容易引入回归问题

#### 现在的方式：红测驱动修 Bug

```
描述 bug 现象 + 相关代码给 AI
        ↓
  AI 写一个"失败测试"（红测）——能复现这个 bug
        ↓
  运行测试，确认测试确实失败（证明 bug 被复现）
        ↓
  AI 修改代码修复 bug
        ↓
  再跑测试，测试通过（变绿）——证明 bug 被修复
```

**具体操作（Prompt 示例）：**

```
## 第一步：写红测复现问题

发现了一个 bug：
- 现象：[描述 bug 的具体表现]
- 复现步骤：[操作步骤]
- 期望行为：[应该是什么样]

相关代码：
[粘贴相关代码]

请先写一个测试用例来复现这个问题。
测试应该能跑通（证明 bug 存在），先不要修改业务代码。
```

```
## 第二步：确认红测失败后，再修复

测试已经能复现问题了。请修改业务代码来修复这个 bug。
修复后，之前的测试应该能通过。
注意：不要修改测试用例来适配代码，而是修改代码来通过测试。
```

#### 为什么这样做

| 传统方式 | 红测驱动 |
|---------|---------|
| 凭经验猜原因 | 用测试精确定位问题 |
| 改完不确定修好没 | 测试通过 = 确认修复 |
| 可能引入新问题 | 测试持续 guarding，防止回归 |
| 改了半天发现改错地方了 | 先复现，方向不会错 |

> **一句话总结**：先证明 bug 存在，再证明 bug 消失了。

**特别适合的场景：**
- 间歇性 bug（不稳定复现的）
- 边界条件导致的 bug（空值、特殊字符、并发）
- 重构后出现的回归 bug
- 复杂逻辑中的计算错误

### 2. 代码重构和优化

```
这段代码性能不好/可读性差，请帮我重构。
要求：
- 保持功能不变
- 提升可读性
- 不引入新的依赖
```

### 3. 知识查询

不用离开编辑器去查文档：

```
React useEffect 的 cleanup 函数什么时候执行？
Tailwind 怎么实现一个 sticky header？
```

## 五、实际效果和心得（5 分钟）

### 效果

用数据说话（可根据实际情况调整）：

- **需求开发效率**：中等复杂度的需求，从接手到提测，整体提速约 40-50%
- **单测覆盖率**：从之前的"基本不写"到现在的"基本都写"
- **bug 率**：没有明显上升（因为有人工 Review 兜底）

### 最大的感受

1. **角色转变**：从"代码编写者"变成"方案评审者+代码审查者"
2. **不是所有需求都适合 AI 做**：复杂交互、特殊动画、性能优化还是需要自己把控
3. **Prompt 质量决定输出质量**：你描述得越清楚，AI 给的方案越好
4. **一定要验证**：AI 生成的代码 80% 是对的，但那 20% 的错可能就是线上事故

---

## 六、对团队的启示（3 分钟）

### 两种团队的区别

**❌ 没有验证环节的团队：**

```
AI 写代码 → 看一眼能跑 → 直接上线
```

结果：bug 增多，代码质量下降，技术债快速积累

**✅ 有验证环节的团队：**

```
AI 提效 → 人做最终把关 → 既快又稳
```

结果：效率提升，质量不降

### 几条实用建议

1. **先出方案再写代码**——永远不要让 AI 直接写代码，先让它告诉你打算怎么改
2. **给足上下文**——代码文件、需求文档、项目约定，信息越多越好
3. **人始终是最后一道关**——AI 可以帮你加速，但不能帮你负责

---

## 七、总结（2 分钟）

### 一句话概括我的做法

> **我不是在用 AI 写代码，我是在"管理 AI 写代码"**

### 完整流程回顾

```
给需求（上下文充分）→ AI 出方案 → 人 Review 方案 → AI 写代码 → AI 写测试 → 人 Review + 验证
```

### 最后的金句

> AI 让开发变快，但"人"决定代码能不能上线。

```
我需要实现 [功能描述]。

需求背景：
- [背景说明]
- [用户场景]

详细需求：
- [具体需求点 1]
- [具体需求点 2]
- ...

相关代码文件：
- [文件路径 1]
- [文件路径 2]

请先阅读这些代码，理解现有架构，然后给出修改方案。
不要写代码，只需要告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 需要注意的边界情况和潜在风险
```

### 模板 2：写代码

```
方案已确认，请按照以下方案编写代码。

注意事项：
- 遵循项目现有代码风格
- 复用项目中已有的工具函数和组件

- 不要引入新的第三方依赖（除非必要且说明理由）

```
请为以下代码编写单元测试。

测试要求：
- 覆盖正常流程
- 覆盖边界情况（空数据、异常输入、特殊字符）
- 覆盖异常处理（接口失败、loading/error 状态）
- 使用项目现有的测试框架和工具
```

### 模板 4：修 Bug（红测驱动）

```
## 第一步：写红测

发现了一个 bug：

- 期望行为：[应该是什么样]

相关代码：

请先写一个测试用例来复现这个问题。
测试应该能跑通（证明 bug 存在），先不要修改业务代码。
```

```
## 第二步：修复

测试已经能复现问题了。请修改业务代码来修复这个 bug。
