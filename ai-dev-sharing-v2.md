## 一、Vibe Coding 是什么？（3 分钟）

### 这个词从哪来的

2025 年 2 月，OpenAI 联合创始人、前 Tesla AI 负责人 **Andrej Karpathy** 发了一条推文：

> *"I just vibe coding — fully give in to the vibes, embrace exponentials, and forget that the code even exists."*

翻译过来就是：**不要管代码怎么写，用自然语言告诉 AI 你要什么，让 AI 去写。**

这个词在 2025 年被 Collins 词典评为**年度词汇**。

### 现在已经不只是"潮流"了

几个数据（来源：GitHub Octoverse 2025、Sonar 调查）：

- **92%** 的美国开发者每天都在用 AI 编程工具
- **46%** 的新代码由 AI 生成
- **59%** 的开发者同时在用 3 个以上的 AI 工具

> 不是"要不要用 AI"，而是"怎么用 AI"。

### 但是——Vibe Coding 也带来了新问题

- AI 生成的代码**重复率上升了 4 倍**（GitClear 2024 报告）
- 开发者对 AI 代码的**信任度从 77% 降到了 60%**（2024 → 2026）
- 安全漏洞、技术债、边界条件遗漏，都是常见问题

所以 Karpathy 自己在 2026 年又提了一个新概念：**Agentic Engineering**——从"跟着感觉走"升级到"有工程纪律的 AI 开发"。

> **今天我想分享的，就是我在实际项目中，怎么把 Vibe Coding 做成一套有纪律的开发流程。**

### 第一步：上下文工程——给 AI 正确的信息

这是很多人忽略的环节，也是决定 AI 输出质量的**最关键因素**。

2025 年行业共识已经从"Prompt Engineering"转向了 **"Context Engineering"**：

```
Prompt Engineering：怎么措辞让 AI 回答更好
Context Engineering：怎么组织信息让 AI 理解更准确
```

#### 我会准备四层上下文：

| 层级 | 内容 | 作用 |
|------|------|------|
| 需求层 | 需求描述 + 产品文档（PRD） | 告诉 AI "做什么" |
| 架构层 | 相关代码文件、目录结构 | 告诉 AI "现有系统长什么样" |
| 规范层 | 代码风格、技术栈约定 | 告诉 AI "遵循什么规则" |
| 约束层 | 不能做什么、需要注意什么 | 防止 AI 踩坑 |

**实际操作示例（Prompt）：**

```
我需要实现一个 XXX 功能。

## 需求背景
- 用户可以在 XXX 页面进行 XXX 操作
- 需要支持 XXX 和 XXX 两种场景
- 详细需求文档见附件

## 约束条件
- 使用项目现有的组件库，不要引入新 UI 库
- 接口需要兼容现有的错误处理中间件
- 移动端需要适配

请先阅读以下代码文件，理解现有架构，然后给我一个修改方案。
不要写代码，只告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 有没有需要注意的边界情况或风险
```

#### 关于上下文的几个实战经验

**1. 信息放对位置**

研究表明，AI 模型对上下文的开头和结尾关注度最高，中间的内容容易被忽略（称为 "Lost in the Middle" 问题）。

所以：
- **开头放**：核心指令、关键约束
- **中间放**：代码文件、参考信息
- **结尾放**：具体要求、输出格式

**2. 上下文不是越多越好**

无关信息会干扰 AI 的判断。只提供和当前任务直接相关的文件和文档。

**3. 迭代式补充**

第一次给的信息不可能完美。AI 出方案后，你发现它遗漏了什么，补充进去再让它重新出方案。这是一个**对话过程**，不是一次性指令。

---

### 第二步：AI 阅读代码，输出修改方案

AI 会返回类似这样的内容：

```
## 修改方案

### 1. 需要修改的文件
- src/pages/OrderList/index.tsx — 新增筛选条件
- src/services/order.ts — 新增接口调用
- src/types/order.d.ts — 新增类型定义

### 2. 修改思路
- OrderList 页面：在现有筛选栏基础上新增一个下拉选择器...
- order service：新增 fetchOrderFilter 接口...
- 类型定义：补充 FilterOption 类型...

### 3. 注意事项
- 现有筛选逻辑用了 URL 参数同步，新增筛选条件也要保持一致
- 下拉数据需要做缓存，避免重复请求
- 移动端需要考虑下拉选择器的交互适配
```

> **关键点**：先出方案，不写代码。方案阶段修改成本最低。

---

### 第三步：人工 Review 方案（最重要的环节）

**这是整个流程里最关键的步骤，也是区分"Vibe Coding"和"Agentic Engineering"的分界线。**

原教旨的 Vibe Coding 是：告诉 AI 你要什么 → AI 直接写 → 看看能不能跑。

而工程化的做法是：**在 AI 动手之前，先确认它的方向是对的。**

我会检查：

- **修改范围**：有没有漏掉文件？有没有改了不该改的？
- **架构一致性**：方案是否符合项目现有的架构规范？
- **遗漏风险**：它提到的注意事项够不够？有没有明显的盲区？
- **更好的方案**：有没有比它说的更简单、更优雅的实现方式？

> **AI 出方案几秒钟，但方案对不对，只有人能判断。**

如果方案有问题，我会补充信息让 AI 重新出方案，而不是让它带着错误方向去写代码。

---

### 第四步：AI 写代码

方案确认后，让 AI 开始写代码：

```
方案确认，请按照上面的方案开始写代码。
注意：
- 遵循项目现有的代码风格
- 使用项目中已有的工具函数和组件
- TypeScript 类型要完整
- 不要引入新的第三方依赖
```

这个阶段我会关注：
- 代码风格是否和项目一致
- 有没有引入不必要的依赖（AI 容易"自作主张"引入新库）
- 命名是否规范
- 有没有重复造轮子

#### 一个重要心态：代码是可丢弃的

Vibe Coding 带来的一个思维转变：

> **代码不再是"精心雕琢的作品"，而是"快速验证想法的材料"。**

AI 生成的代码不满意？让它重写。方案走不通？换一个方向让它再来。

以前手写代码改一次成本很高，现在 AI 重新生成几乎是零成本。这意味着你可以**更频繁地尝试不同方案**，而不是第一个方案就凑合用。

---

### 第五步：AI 写测试

代码写完后，让 AI 补充测试：

```
请为新增的代码编写单元测试。
重点关注：
- 正常流程
- 边界情况（空数据、异常输入）
- 异常处理（接口失败、loading 状态）
```

> 之前手动写单测是最容易被跳过的环节，现在 AI 生成测试基本能覆盖 70-80% 的场景。

但要注意：AI 生成的测试容易只覆盖"正常路径"（happy path），边界情况和异常场景需要你额外强调。

---

### 第六步：最终验证——人的兜底（最后一道防线）

**这一步决定了 AI 提效的最终质量。**

行业数据显示：AI 生成的代码中，大约 20% 存在某种问题（安全漏洞、边界遗漏、不符合规范）。如果不验证就上线，就变成了"加速出错"。

我会做三件事：

#### 1. Code Review

标准不是"能不能跑"，而是：

- 是否符合现有代码风格和架构规范
- 是否引入了重复逻辑（AI 代码重复率是人工代码的 4 倍）
- 是否有潜在副作用
- 类型定义是否严谨
- 有没有安全隐患（硬编码密钥、缺少输入校验等）

> **标准不是"能不能跑"，而是"能不能长期维护"**

#### 2. 功能验证

| 验证类型 | 关注点 |
|---------|--------|
| 正常流程 | 核心功能是否完整可用 |
| 边界情况 | 空数据、极端输入、特殊字符 |
| 异常处理 | 接口失败、网络超时、loading/error 状态 |

> **测试用例是"理论覆盖"，手动验证是"真实体验"**

#### 3. 体验检查

这个是 AI 最容易忽略的部分：

- 交互是否流畅，有没有卡顿
- loading 状态展示是否合理
- 错误提示是否用户友好
- 各种屏幕尺寸下布局是否正常

---

### 这个流程的核心价值

```
AI 可以保证"生成"，但只有人能保证"正确 + 合理 + 可用"
```

```
没有最后这一步，前面所有提效都有可能变成"加速出错"
```

## 四、修 Bug：红测驱动（5 分钟）

开发新需求和修 Bug 是两种不同的工作模式。修 Bug 我有一套单独的流程：

### 核心原则

> **先让 bug 能被稳定复现，再动手改代码**

### 流程

```
描述 bug 现象 + 相关代码给 AI
        ↓
  AI 写一个"失败测试"（红测）——能复现这个 bug
        ↓
  运行测试，确认测试确实失败（证明 bug 被复现）
        ↓
  AI 修改代码修复 bug
        ↓
  再跑测试，测试通过（变绿）——证明 bug 被修复
```

### 为什么这样做

| 传统方式 | 红测驱动 |
|---------|---------|
| 凭经验猜原因 | 用测试精确定位问题 |
| 改完不确定修好没 | 测试通过 = 确认修复 |
| 可能引入新问题 | 测试持续 guarding，防止回归 |
| 改了半天发现改错地方了 | 先复现，方向不会错 |

**Prompt 示例：**

```
## 第一步：写红测

发现了一个 bug：
- 现象：[描述 bug 的具体表现]
- 复现步骤：[操作步骤]
- 期望行为：[应该是什么样]

相关代码：
[粘贴相关代码]

请先写一个测试用例来复现这个问题。
先不要修改业务代码。
```

```
## 第二步：修复

测试已经能复现问题了。请修改业务代码来修复这个 bug。
注意：不要修改测试用例来适配代码，而是修改代码来通过测试。
```

> **一句话总结**：先证明 bug 存在，再证明 bug 消失了。

## 五、实际效果和踩过的坑（5 分钟）

### 效果

- **需求开发效率**：中等复杂度的需求，整体提速约 40-50%
- **单测覆盖率**：从"基本不写"到"基本都写"
- **Bug 率**：没有明显上升（因为有人工 Review 兜底）

### 踩过的坑（这些才是真经验）

**1. AI 会"自作聪明"**

- 引入你没用过的第三方库来解决问题
- 用一种看起来高级但项目里没人用的写法
- 重构你没让它重构的代码

**解法**：在 Prompt 里明确约束——"不要引入新依赖"、"只修改我指定的文件"。

**2. 上下文不足导致方案跑偏**

- AI 不了解你的业务逻辑，可能提出技术上正确但业务上错误的方案
- AI 不知道历史遗留问题，可能踩到已知的坑

**解法**：Review 方案时重点关注业务逻辑正确性，不只是技术实现。

**3. 测试的"虚假安全感"**

- AI 生成的测试可能只覆盖了 happy path
- 测试名字看起来很全，但实际断言不够严格
- Mock 太多，测试通过了但实际场景不一定对

**解法**：不要只看测试通过率，要 Review 测试内容本身。

**4. 代码重复问题**

- AI 在不同地方生成类似的逻辑，不知道项目里已经有工具函数
- 导致代码库膨胀，后续维护困难

**解法**：在上下文中提供已有的工具函数列表，或者让 AI 在写代码前先搜索项目中是否已有类似实现。

## 六、什么适合 AI 做，什么不适合（3 分钟）

这是一个被很多人忽略但非常重要的判断。

### 适合 AI 做的

| 场景 | 原因 |
|------|------|
| 标准 CRUD 页面 | 模式固定，AI 很擅长 |
| 表单 + 校验 | 规则明确，AI 覆盖全面 |
| 对接 API + 类型定义 | 重复性高，AI 效率高 |
| 写单测 | AI 生成测试几乎零成本 |
| 代码重构 | AI 可以逐文件处理，不遗漏 |
| Bug 定位和修复 | 结合红测，效果很好 |

### 不太适合完全交给 AI 的

| 场景 | 原因 |
|------|------|
| 复杂交互和动画 | 需要反复调试体验，AI 的"直觉"不够 |
| 性能优化 | 需要深入理解运行时行为，AI 容易给出表面方案 |
| 核心业务逻辑 | AI 不理解业务上下文，容易出错 |
| 安全相关代码 | AI 容易忽略安全细节，必须人工把关 |

> **原则**：AI 负责模式化、重复性的工作；人负责需要判断力和创造力的工作。

## 七、对团队的建议（3 分钟）

### 两类团队的区别

**❌ 无纪律的 Vibe Coding：**

```
描述需求 → AI 直接写代码 → 看一眼能跑 → 上线
```

结果：Bug 增多，代码质量下降，技术债快速积累，线上事故频发

**✅ 有工程纪律的 AI 开发：**

```
准备上下文 → AI 出方案 → 人审方案 → AI 写代码 → AI 写测试 → 人验证
```

结果：效率提升，质量不降，可持续

### 几条实用建议

**1. 永远先出方案再写代码**

不要让 AI 直接写代码。方案阶段的修改成本接近零，代码写完再改成本就高了。

**2. 上下文是第一优先级**

花 5 分钟整理好上下文（需求、代码、约束），比花 30 分钟修 AI 跑偏的代码值。

**3. 人是最后一道关**

AI 可以帮你加速，但不能帮你负责。每一行上线的代码，最终都是人在负责。

**4. 区分场景用 AI**

不是所有代码都适合让 AI 写。标准化的工作交给 AI，需要创造力和判断力的工作留给自己。

**5. 建立团队约定**

- 统一工具和流程
- AI 生成的代码必须经过 Code Review
- 关键模块和核心逻辑建议人工编写

## 八、总结（2 分钟）

### 三个层次的理解

```
第一层：用 AI 写代码（工具层）——让 AI 帮你写函数、写页面
第二层：用 AI 管开发流程（方法层）——先方案后代码，红测驱动修 Bug
第三层：重新定义开发者角色（认知层）——从"写代码的人"变成"管理 AI 的人"
```

### 一句话概括

> **我不是在用 AI 写代码，我是在"管理 AI 写代码"**

### 最后引用 Karpathy 的话

> *"You can outsource your typing to AI, but you can never outsource your understanding."*

## 附录：Prompt 模板

### 模板 1：出方案（带上下文结构）

```
我需要实现 [功能描述]。

## 需求背景
- [背景说明]
- [用户场景]

## 详细需求
- [具体需求点 1]
- [具体需求点 2]

## 约束条件
- [技术约束 1]
- [业务约束 2]

## 相关代码
- [文件路径 1]
- [文件路径 2]

请先阅读这些代码，理解现有架构，然后给出修改方案。
不要写代码，只需要告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 需要注意的边界情况和潜在风险
```

### 模板 2：写代码

```
方案已确认，请按照上面的方案编写代码。

约束条件：
- 遵循项目现有代码风格
- 复用项目中已有的工具函数和组件
- TypeScript 类型要完整
- 不要引入新的第三方依赖（除非必要且说明理由）
- 只修改方案中涉及的文件，不要改动其他文件
```

### 模板 3：写测试

```
请为新增的代码编写单元测试。

测试要求：
- 覆盖正常流程
- 覆盖边界情况（空数据、异常输入、特殊字符）
- 覆盖异常处理（接口失败、loading/error 状态）
- 不要只测 happy path，重点测异常场景
- 使用项目现有的测试框架和工具
```

### 模板 4：修 Bug（红测驱动）

```
## 第一步：写红测

发现了一个 bug：
- 现象：[描述 bug 的具体表现]
- 复现步骤：[操作步骤]
- 期望行为：[应该是什么样]

相关代码：
[粘贴相关代码]

请先写一个测试用例来复现这个问题。
先不要修改业务代码。
```

```
## 第二步：修复

测试已经能复现问题了。请修改业务代码来修复这个 bug。
修复后，之前的测试应该能通过。
注意：不要修改测试用例来适配代码，而是修改代码来通过测试。
```
