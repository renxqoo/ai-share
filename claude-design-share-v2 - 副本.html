Why Not Start Coding
3 / 18

为什么不能让 AI 一上来就写代码？

01

结构很快失控

一上来就写，很容易先产出局部代码，再慢慢偏离整体设计。

02

反复纠偏返工

需要不断补充背景、修正方向，返工成本往往高于第一次写慢一点。

03

功能能跑但质量差

代码能工作，不代表结构合理、边界完整、可测试、可维护。

04

最后还得重构

短期省掉的思考，常常在后期以更大的维护成本补回来。

最大的问题不是写不出来，而是写出来以后越来越难收拾。

Agent Principle

AI Agent 实现原理

为什么 AI 能帮你干活？

LLM （大模型） + TOOLS （工具） + LOOP （循环执行） = AI Agent

它看起来像在“自己干活”，本质上是因为它会结合上下文反复判断，并在需要时调用工具。

Agent Loop
10 / 18

Agent 循环

接收任务
用户说一句话
LLM 思考
这件事要怎么做
Decision
LLM 决定
自己能回答
直接输出结果
如果信息已经足够，模型就直接整理答案并结束。
需要工具
决定调用哪个工具，传什么参数
系统执行工具
→
结果返回 LLM
→
继续下一轮判断
这就是为什么 Agent 看起来像会“自己继续做下去”。
Agent 的本质不是一次生成答案，而是“判断 → 调用工具 → 看结果 → 再判断”的循环。
Execution Mechanism
11 / 18

Agent 的执行机制

pseudo.tswhile (!taskComplete) { // 基于当前上下文和可用工具，判断下一步该做什么 const response = LLM(messages, tools); // 如果模型认为需要调用工具 if (response.toolCalls) { // 系统执行工具调用 const results = executeTools(response.toolCalls); // 把工具结果放回上下文，进入下一轮判断 messages.push(...results); } else { // 如果不需要工具，直接输出最终结果 return response.content; } }
第 1 轮
用户：帮我查一下杭州今天的天气
LLM 判断：这是实时信息，不能直接编
动作：调用天气工具
第 2 轮
工具返回：杭州，晴，25°C，东北风 2 级
LLM 判断：信息已经够了
动作：整理结果，准备回复
第 3 轮
输出最终答案
循环结束
Agent 不是一次想完答案，而是带着最新上下文反复判断和执行。
Messages & Tools
12 / 18

messages 和 tools 是什么？

messages[ { "role": "system", "content": "你是个人 AI 助手，需要理解用户的问题，并结合上下文和工具能力帮助用户解决问题。" }, { "role": "user", "content": "帮我查一下杭州今天的天气" }, { "role": "assistant", "tool_calls": [ { "id": "call_1", "type": "function", "function": { "name": "get_weather", "arguments": "{\"city\":\"杭州\"}" } } ] }, { "role": "tool", "tool_call_id": "call_1", "content": "杭州，晴，25°C，东北风 2 级" }, { "role": "assistant", "content": "杭州今天是晴天，25°C，东北风 2 级。" } ]
tools[ { // 工具类型，这里表示函数调用 "type": "function", "function": { // 工具名称，模型会用它决定调用哪个工具 "name": "get_weather", // 工具作用说明 "description": "查询指定城市的天气信息", // 工具参数定义 "parameters": { "type": "object", "properties": { "city": { // 参数类型 "type": "string", // 参数说明 "description": "城市名称，例如：杭州" } }, // 必填参数 "required": ["city"] } } } ]
messages 决定它“看到了什么”，tools 决定它“能做什么”。

Business Direction

我们能不能给 Agent 一套更好用的操作方式？

既然 AI Agent 已经能处理信息和任务
那我们公司的项目，能不能也提供一种更适合 Agent 操作的方式？
这就是后面这一部分想展开的方向。
不是重做系统，而是换一种更适合 Agent 操作的方式。
CLI Layer
16 / 18

思路：把业务操作变成 CLI 命令

Business CLI saas shell
$ saas payment list --status=pending $ saas payment detail --id=PAY20260509001 $ saas deal quote --pair=USD/CNH $ saas approval list --pending $ saas balance check --currency=USD
CLI 不是重做一套系统，而是给现有系统再加一层适合 Agent 调用的壳。
Business Agent Flow

Agent 怎么接手这些业务动作？

Round 2

你：这笔交易的汇率是多少？
