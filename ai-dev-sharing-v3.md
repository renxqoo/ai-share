## 一、Vibe Coding：从潮流到工程（5 分钟）

### 这个词从哪来的

2025 年 2 月，OpenAI 联合创始人 **Andrej Karpathy** 发了一条推文：

> *"Vibe coding — fully give in to the vibes, embrace exponentials, and forget that the code even exists."*

翻译：**不要管代码怎么写，用自然语言告诉 AI 你要什么，让 AI 去写。**

2025 年底，这个词被 Collins 词典评为**年度词汇**。

### 2026 年的真实状态

不是"要不要用"，而是"怎么用"。几个最新数据：

| 数据 | 来源 |
|------|------|
| **92%** 的美国开发者每天用 AI 编程工具 | JetBrains 2025 Q4 |
| **41%** 的全球代码由 AI 生成 | JetBrains 2026 |
| **87%** 的财富 500 强运行至少一个 AI 编程平台 | Industry Report 2026 |
| **80%** 的财富 500 强有 AI Agent 在生产环境运行 | Enterprise Survey 2026 |
| AI 编程市场规模：2025 年 **$47 亿** → 2027 年预计 **$123 亿** | Market Analysis |

> **对管理层的一句话**：AI 辅助编程已经不是前沿实验，是行业基础设施。

### 但是——问题也来了

| 问题 | 数据 |
|------|------|
| AI 代码**重复率上升 4 倍** | GitClear 2024 |
| **45%** 的 AI 生成代码包含 OWASP Top-10 安全漏洞 | Security Report 2026 |
| AI 参与的 PR 安全问题率是人工的 **2.74 倍** | Research 2026 |
| **96%** 的开发者不完全信任 AI 生成的代码 | Survey 2026 |

这就是 **"信任悖论"**：96% 的人不完全信任，但还在用。82% 的人承认 AI 让他们更快，但只有 48% 总是在提交前 Review 代码。

> 速度和信任之间有一个巨大的裂缝。

### 从 Vibe Coding 到 Harness Engineering：概念的快速进化

2025-2026 年，AI 开发方法论经历了四次快速迭代：

```
Prompt Engineering (2023)
  怎么措辞让 AI 回答更好
      ↓
Vibe Coding (Feb 2025, Karpathy)
  不管代码，用自然语言告诉 AI 你要什么
      ↓
Context Engineering (Dec 2025, Karpathy)
  不只是措辞，而是组织正确的信息让 AI 理解
      ↓
Agentic Engineering (Feb 2026, Karpathy)
  人来编排 Agent，Agent 来执行，保持专业质量标准
      ↓
Harness Engineering (Feb 2026, Hashimoto → OpenAI)
  设计约束、反馈循环、验证系统——包裹 Agent 的完整控制系统
```

#### Agentic Engineering——Karpathy 的升级

2026 年 4 月，Karpathy 在 Sequoia AI Ascent 大会上明确区分：

```
Vibe Coding（2025）= "降低了门槛"——任何人都能快速构建软件
Agentic Engineering（2026）= "提升了上限"——用工程纪律保证专业质量
```

> *"Agentic engineering is about preserving the quality bar of professional software.
> You are not allowed to introduce vulnerabilities because of vibe coding.
> You are still responsible for your software, just as before."*

#### Harness Engineering——2026 年最新的工程学科

2026 年 2 月，Terraform 创始人 **Mitchell Hashimoto** 提出了一个原则：

> *"Every time you discover an agent has made a mistake, you take the time to engineer a solution so that it can never make that mistake again."*
> （每次发现 Agent 犯错，都要花时间设计一个方案，确保它永远不会再犯同样的错。）

随后 OpenAI 发布了一篇博客，描述了一个小团队如何用 **零行人工代码** 构建了超过 **100 万行**的生产级代码。他们靠的不是更强的模型，而是设计了一套包裹 Agent 的完整控制系统。

LangChain 总结了一个公式：

```
Agent = Model + Harness
```

**模型负责推理，Harness 负责其他一切**：上下文、约束、反馈循环、验证、迭代优化。

这意味着：

```
以前：工程师写代码
现在：工程师设计 Harness（约束系统），Agent 在 Harness 里写代码
```

> **今天我想分享的，就是我在实际项目中做的 Harness Engineering——我为 AI Agent 设计了什么样的约束、反馈循环和验证流程，让它既能快速产出代码，又能保证质量。**

## 三、我的完整工作流（15 分钟，核心环节）

### 整体流程图——我的 Harness 长什么样

```
┌──────────────────────────────────────────────────────────────────────┐
│               Plan → Execute → Verify（PEV 框架）                    │
│                                                                      │
│  ┌─ PLAN ──────────────────────────────────────────────────┐        │
│  │  1. 准备上下文 ──→ 2. AI 出方案 ──→ 3. 人工 Review 方案  │        │
│  └──────────────────────────────────┬───────────────────────┘        │
│                                     │ 确认无误                       │
│  ┌─ EXECUTE ────────────────────────↓───────────────────────┐       │
│  │  4. AI 写代码 ──→ 5. AI 写测试（含自动修复循环）           │       │
│  └──────────────────────────────────┬───────────────────────┘        │
│                                     │                                │
│  ┌─ VERIFY ─────────────────────────↓───────────────────────┐       │
│  │  6. 人工 Code Review + 功能验证 + 体验检查                 │       │
│  └──────────────────────────────────┬───────────────────────┘        │
│                                     ↓                                │
│                                  提测上线                            │
└──────────────────────────────────────────────────────────────────────┘
```

这个结构对应了 2026 年行业共识的 **PEV 框架**（Plan-Execute-Verify），也是 Harness Engineering 的核心实践：

| 阶段 | 人的角色（Harness 设计者） | AI 的角色 |
|------|---------|----------|
| Plan（计划） | 给上下文、审方案 | 读代码、出方案 |
| Execute（执行） | 监督方向 | 写代码、写测试、自修复 |
| Verify（验证） | 最终审查 | 辅助检查（lint、测试） |

> 关键理念：**AI 负责执行，人负责设计和验证 Harness**。两头是人，中间是 AI。

> Hashimoto 原则：每次发现 AI 犯错，就把防错机制写进 Harness（比如 CLAUDE.md 里加一条规则），让它永远不会再犯同样的错。

### 第一步：上下文工程——比 Prompt Engineering 更重要

2026 年行业已经形成共识：**Context Engineering 正在取代 Prompt Engineering**。

```
Prompt Engineering：怎么措辞让 AI 回答更好
Context Engineering：怎么组织信息让 AI 理解更准确
```

一个关键数据：**82% 的 IT/数据负责人认为单靠 Prompt Engineering 已经不够了**。

#### 上下文的两层架构

**第一层：持久化上下文（跨会话生效）**

这是 2026 年最大的变化之一——用项目级配置文件让 AI "记住" 你的项目规范：

| 文件 | 作用 |
|------|------|
| `CLAUDE.md` | Claude Code 的项目级指令：技术栈、代码风格、约定 |
| `.cursorrules` | Cursor 的项目级规则：组件规范、命名约定 |
| `.github/copilot-instructions.md` | Copilot 的自定义指令 |

**实际效果**：写好这些文件后，AI 每次启动都自动知道你的项目规范，不用每次重复说明。**这比换一个更强的模型更能提升输出质量。**

> 比喻：持久化上下文就像"新人入职手册"——AI 不需要你每次都从头介绍项目。

**第二层：任务上下文（每次任务准备）**

| 层级 | 内容 | 作用 |
|------|------|------|
| 需求层 | 需求描述 + 产品文档（PRD） | 告诉 AI "做什么" |
| 架构层 | 相关代码文件、目录结构 | 告诉 AI "现有系统长什么样" |
| 规范层 | 代码风格、技术栈约定 | 告诉 AI "遵循什么规则" |
| 约束层 | 不能做什么、需要注意什么 | 防止 AI 踩坑 |

**实际操作示例（Prompt）：**

```
我需要实现一个 XXX 功能。

## 需求背景
- 用户可以在 XXX 页面进行 XXX 操作
- 需要支持 XXX 和 XXX 两种场景
- 详细需求文档见附件

## 约束条件
- 使用项目现有的组件库，不要引入新 UI 库
- 接口需要兼容现有的错误处理中间件
- 移动端需要适配

请先阅读以下代码文件，理解现有架构，然后给我一个修改方案。
不要写代码，只告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 有没有需要注意的边界情况或风险
```

#### 上下文管理的实战经验

**1. 信息放对位置**

研究表明 AI 对上下文的开头和结尾关注度最高（"Lost in the Middle" 问题）。

```
开头 → 核心指令、关键约束
中间 → 代码文件、参考信息
结尾 → 具体要求、输出格式
```

**2. 上下文不是越多越好**

无关信息会干扰 AI 判断。只提供和当前任务**直接相关**的文件。

**3. MCP：上下文的新范式**

2026 年一个重大变化是 **MCP（Model Context Protocol）**——Anthropic 发布的开放协议，现在已捐赠给 Linux Foundation，28% 的财富 500 强已部署。

```
没有 MCP：每个 AI 工具各自集成，GitHub 在 Cursor 和 Copilot 里是两套
有了 MCP：一个 MCP Server 连 GitHub，所有 AI 工具共享同一个接口
```

类比：**MCP 让 AI 从"你喂什么吃什么"变成"自己知道去哪里找信息"**——就像从"背着一整个图书馆"变成"知道怎么去图书馆查资料"。

### 第二步：AI 阅读代码，输出修改方案

AI 会返回类似这样的内容：

```
## 修改方案

### 1. 需要修改的文件
- src/pages/OrderList/index.tsx — 新增筛选条件
- src/services/order.ts — 新增接口调用
- src/types/order.d.ts — 新增类型定义

### 2. 修改思路
- OrderList 页面：在现有筛选栏基础上新增一个下拉选择器...
- order service：新增 fetchOrderFilter 接口...
- 类型定义：补充 FilterOption 类型...

### 3. 注意事项
- 现有筛选逻辑用了 URL 参数同步，新增筛选条件也要保持一致
- 下拉数据需要做缓存，避免重复请求
- 移动端需要考虑下拉选择器的交互适配
```

> **关键点**：先出方案，不写代码。方案阶段修改成本最低。

---

### 第三步：人工 Review 方案（最重要的环节）

**这是区分 Vibe Coding 和 Agentic Engineering 的分界线。**

Forbes 2026 年 4 月的一篇文章提出了一个概念叫 **"决策压缩"（Decision Compression）**：

> 当从想法到成品的周期从几个月压缩到几小时，所有质量控制环节——设计评审、安全审查、代码审查——都会被绕过。

这正是我在这个环节要防止的。

我会检查：

- **修改范围**：有没有漏掉文件？有没有改了不该改的？
- **架构一致性**：方案是否符合项目现有的架构规范？
- **业务正确性**：AI 不懂你的业务，可能提出技术正确但业务错误的方案
- **安全意识**：AI 容易忽略安全细节（45% 的 AI 代码含安全漏洞不是开玩笑的）
- **更好的方案**：有没有比它说的更简单、更优雅的实现方式？

> **AI 出方案几秒钟，但方案对不对，只有人能判断。**

---

### 第四步：AI 写代码

方案确认后，让 AI 开始写代码：

```
方案确认，请按照上面的方案开始写代码。
约束条件：
- 遵循项目现有代码风格
- 使用项目中已有的工具函数和组件
- TypeScript 类型要完整
- 不要引入新的第三方依赖
- 只修改方案中涉及的文件，不要改动其他文件
```

#### 一个重要心态：代码是可丢弃的

> **代码不再是"精心雕琢的作品"，而是"快速验证想法的材料"。**

不满意？让它重写。方案走不通？换个方向再来。以前手写代码改一次成本很高，现在重新生成几乎零成本。

这意味着你可以**更频繁地尝试不同方案**，而不是第一个方案就凑合用。

Karpathy 甚至用 AI 构建"一次性应用"来定位单个 bug——代码用完即丢，因为生成成本已经趋近于零。

---

### 第五步：AI 写测试（含自动修复循环）

代码写完后，让 AI 补充测试：

```
请为新增的代码编写单元测试。
重点关注：
- 正常流程
- 边界情况（空数据、异常输入）
- 异常处理（接口失败、loading 状态）
- 不要只测 happy path，重点测异常场景
```

2026 年的 Agent 工具（如 Claude Code、Cursor）支持**自动修复循环**：

```
AI 写测试 → 运行测试 → 测试失败 → AI 自动分析失败原因 → AI 自动修复 → 重新运行 → 直到通过
```

> 之前手动写单测是最容易被跳过的环节，现在 AI 生成 + 自动修复，测试覆盖率从"基本不写"变成了"基本都写"。

**但要注意**：AI 生成的测试容易只覆盖 happy path，Mock 太多时测试通过但实际场景不一定对。不要只看通过率，要 Review 测试内容本身。

---

### 第六步：最终验证——人的兜底（最后一道防线）

**这一步决定了 AI 提效的最终质量。**

行业数据很明确：

```
96% 的开发者不完全信任 AI 代码
45% 的 AI 代码包含安全漏洞
AI 参与的 PR 安全问题率是人工的 2.74 倍
```

如果不验证就上线，就不是"提效"，是"加速出错"。

#### 1. Code Review

- 是否符合现有代码风格和架构规范
- 是否引入了重复逻辑（AI 代码重复率是人工的 4 倍）
- 有没有安全隐患（硬编码密钥、缺少输入校验、SQL 注入等）
- 类型定义是否严谨
- 有没有潜在的副作用

> **标准不是"能不能跑"，而是"能不能长期维护"**

#### 2. 功能验证

| 验证类型 | 关注点 |
|---------|--------|
| 正常流程 | 核心功能是否完整可用 |
| 边界情况 | 空数据、极端输入、特殊字符 |
| 异常处理 | 接口失败、网络超时、loading/error 状态 |

#### 3. 体验检查（AI 最容易忽略的）

- 交互是否流畅，有没有卡顿
- loading 状态展示是否合理
- 错误提示是否用户友好
- 各种屏幕尺寸下布局是否正常

---

### 这个流程的核心价值

```
AI 可以保证"生成"，但只有人能保证"正确 + 合理 + 可用"
```

```
速度没有判断力，就是负债。——Forbes 2026
```

## 四、修 Bug：红测驱动（5 分钟）

开发新需求和修 Bug 是两种不同的工作模式。修 Bug 我有一套单独的流程：

### 核心原则

> **先让 bug 能被稳定复现，再动手改代码**

### 流程

```
描述 bug 现象 + 相关代码给 AI
        ↓
  AI 写一个"失败测试"（红测）——能复现这个 bug
        ↓
  运行测试，确认测试确实失败（证明 bug 被复现）
        ↓
  AI 修改代码修复 bug
        ↓
  再跑测试，测试通过（变绿）——证明 bug 被修复
```

### 为什么这样做

| 传统方式 | 红测驱动 |
|---------|---------|
| 凭经验猜原因 | 用测试精确定位问题 |
| 改完不确定修好没 | 测试通过 = 确认修复 |
| 可能引入新问题 | 测试持续 guarding，防止回归 |
| 改了半天发现改错地方了 | 先复现，方向不会错 |

> **一句话总结**：先证明 bug 存在，再证明 bug 消失了。

## 五、实际效果和踩过的坑（5 分钟）

### 效果

- **需求开发效率**：中等复杂度的需求，整体提速约 40-50%
- **单测覆盖率**：从"基本不写"到"基本都写"
- **Bug 率**：没有明显上升（因为有人工 Review 兜底）

### 踩过的坑（这些才是真经验）

**每一个坑，都是一次改进 Harness 的机会。** 按照 Hashimoto 原则——每发现一个错误，就设计一个机制防止它再发生。

**1. AI 会"自作聪明"**

- 引入你没用过的第三方库来解决问题
- 用一种看起来高级但项目里没人用的写法
- 重构你没让它重构的代码

**Harness 解法**：在持久化上下文文件（CLAUDE.md / .cursorrules）里明确约定禁止事项，在 Prompt 里再次强调。

**2. 上下文不足导致方案跑偏**

- AI 不了解业务逻辑，可能提出技术正确但业务错误的方案
- AI 不知道历史遗留问题，可能踩到已知的坑

**Harness 解法**：Review 方案时重点关注**业务逻辑正确性**，不只是技术实现。把常见的业务规则写进 CLAUDE.md。

**3. 测试的"虚假安全感"**

- AI 生成的测试可能只覆盖 happy path
- Mock 太多，测试通过了但实际场景不一定对

**Harness 解法**：Review 测试内容本身，不要只看通过率。在 Prompt 中明确要求"重点测异常场景"。

**4. 代码重复问题**

- AI 在不同地方生成类似逻辑，不知道项目里已有工具函数

**Harness 解法**：在 CLAUDE.md 中列出已有的工具函数，或者让 AI 在写代码前先搜索项目中是否已有类似实现。

**5. 安全盲区**

- AI 代码的安全漏洞率远高于人工代码
- 容易忽略输入校验、权限检查、敏感信息处理

**Harness 解法**：CLAUDE.md 中加入安全规则清单（如"所有用户输入必须校验"），Code Review 时增加安全检查项，考虑引入自动安全扫描（Semgrep / Snyk）作为 CI 质量门禁。

## 六、什么适合 AI 做，什么不适合（3 分钟）

### 适合 AI 做的

| 场景 | 原因 |
|------|------|
| 标准 CRUD 页面 | 模式固定，AI 很擅长 |
| 表单 + 校验 | 规则明确，AI 覆盖全面 |
| 对接 API + 类型定义 | 重复性高，AI 效率高 |
| 写单测 | AI 生成测试几乎零成本 |
| 代码重构 | AI 可以逐文件处理，不遗漏 |
| Bug 定位和修复 | 结合红测，效果很好 |
| 快速原型 | 代码可丢弃，试错成本低 |

### 不太适合完全交给 AI 的

| 场景 | 原因 |
|------|------|
| 复杂交互和动画 | 需要反复调试体验，AI 的"直觉"不够 |
| 性能优化 | 需要深入理解运行时行为 |
| 核心业务逻辑 | AI 不理解业务上下文 |
| 安全相关代码 | 45% 漏洞率，必须人工把关 |
| 跨系统集成 | 边界条件复杂，AI 容易遗漏 |

> **原则**：AI 负责模式化、重复性的工作；人负责需要判断力和创造力的工作。

## 七、对团队的建议——治理比工具更重要（3 分钟）

### 一个扎心的数据

```
80% 的企业已经在用 AI 编程工具
但只有 6% 有成熟的 AI 安全策略
79% 的组织缺乏成熟的 Agent 治理模型
```

> 大多数团队的现状是：工具跑在治理前面。

### 两类团队的区别

**❌ 无纪律的 Vibe Coding：**

```
描述需求 → AI 直接写代码 → 看一眼能跑 → 上线
```

结果：Bug 增多，安全漏洞累积，技术债快速膨胀，线上事故频发

**✅ 有工程纪律的 AI 开发：**

```
准备上下文 → AI 出方案 → 人审方案 → AI 写代码 → AI 写测试 → 人验证
```

结果：效率提升，质量不降，可持续

### 团队落地建议

**1. 建立持久化上下文**

每个项目维护 `CLAUDE.md` 或 `.cursorrules`，把团队规范写进去。这是一次投入、持续收益的事情。

**2. 永远先出方案再写代码**

不要让 AI 直接写代码。方案阶段修改成本接近零。

**3. AI 生成的代码必须经过 Code Review**

和人工代码同样的 Review 标准，甚至更严格。特别关注安全问题和重复代码。

**4. 区分场景**

不是所有代码都适合让 AI 写。标准化的工作交给 AI，核心逻辑和安全相关留给自己。

**5. 引入自动化质量门禁**

- lint + 类型检查
- 安全扫描（Semgrep / Snyk）
- 单测覆盖率阈值
- PR 中标记 AI 生成的代码

## 八、总结（2 分钟）

### AI 辅助开发的三个层次

```
第一层：用 AI 写代码（工具层）
  ——让 AI 帮你写函数、写页面

第二层：用 AI 管开发流程（方法层）
  ——上下文工程、PEV 框架、红测驱动

第三层：设计 Harness（认知层）
  ——从"写代码的人"变成"设计 Agent 约束系统的人"
```

### 核心公式

```
Agent = Model + Harness
```

模型在飞速进化，但真正决定 AI 代码质量的，是你设计的 Harness。

### 一句话概括

> **我不是在用 AI 写代码，我是在为 AI 设计一个可靠的运行环境**

### 最后

> *"Every time you discover an agent has made a mistake, you take the time to engineer a solution so that it can never make that mistake again."*
> —— Mitchell Hashimoto, 2026

> *"You can outsource your typing to AI, but you can never outsource your understanding."*
> —— Andrej Karpathy, 2026

### C. Prompt 模板

#### 模板 1：出方案（带上下文结构）

```
我需要实现 [功能描述]。

## 需求背景
- [背景说明]
- [用户场景]

## 详细需求
- [具体需求点 1]
- [具体需求点 2]

## 约束条件
- [技术约束 1]
- [业务约束 2]

## 相关代码
- [文件路径 1]
- [文件路径 2]

请先阅读这些代码，理解现有架构，然后给出修改方案。
不要写代码，只需要告诉我：
1. 需要修改哪些文件
2. 每个文件的修改思路
3. 需要注意的边界情况和潜在风险
```

#### 模板 2：写代码

```
方案已确认，请按照上面的方案编写代码。

约束条件：
- 遵循项目现有代码风格（参见 CLAUDE.md）
- 复用项目中已有的工具函数和组件
- TypeScript 类型要完整，不要使用 any
- 不要引入新的第三方依赖
- 只修改方案中涉及的文件
```

#### 模板 3：写测试

```
请为新增的代码编写单元测试。

测试要求：
- 覆盖正常流程
- 覆盖边界情况（空数据、异常输入、特殊字符）
- 覆盖异常处理（接口失败、loading/error 状态）
- 不要只测 happy path，重点测异常场景
- Mock 最小化，优先使用真实逻辑
- 使用项目现有的测试框架（Vitest + Testing Library）
```

#### 模板 4：修 Bug（红测驱动）

```
## 第一步：写红测

发现了一个 bug：
- 现象：[描述 bug 的具体表现]
- 复现步骤：[操作步骤]
- 期望行为：[应该是什么样]

相关代码：
[粘贴相关代码]

请先写一个测试用例来复现这个问题。
先不要修改业务代码。
```

```
## 第二步：修复

测试已经能复现问题了。请修改业务代码来修复这个 bug。
修复后，之前的测试应该能通过。
注意：不要修改测试用例来适配代码，而是修改代码来通过测试。
```

#### 模板 5：代码重构

```
这段代码需要重构，当前问题：
- [性能问题 / 可读性差 / 逻辑重复 / ...]

要求：
- 保持功能不变
- 提升可读性和可维护性
- 不引入新的第三方依赖
- 确保现有测试仍然通过
```
