Risk03 / 18

为什么不能一上来就让 AI 直接写？

1
自信地写错
最危险的不是不会，而是很像会。看起来像正确答案，细节却偏了。
2
重复造轮子
不理解现有代码资产时，最容易把已经存在的能力再写一遍。
3
漏看上下文
改一个文件，影响三个模块。没把上下文带够，输出就会变脆。
4
能跑，但难维护
局部可运行不等于系统可维护。架构、约束、边界都可能被悄悄破坏。
很多时候，问题不是 AI 完全不会做，而是它很容易做得“像对的一样”。
先控风险

Plan05 / 18

PLAN：先出计划，不先写代码

先给 Agent 三样东西
需求描述：你已经整理过的目标和口径
PRD 原文：补足细节、边界、业务语义
约束条件：目录、规范、依赖、风险、时限
任务拆解
先拆成可执行块，再决定谁做什么。
关键文件
明确入口、依赖点、影响范围，不盲改。
架构权衡
把“这样做的代价”先说出来，而不是事后补锅。
实施步骤
先后顺序清楚，验证点清楚，才能放行。
先看清方向，再让 AI 动手。
Spec First
Release06 / 18

确认方案后，再放行执行

先审查计划
步骤是否合理 场景是否遗漏 范围是否清楚
再给反馈（如果需要）
提出疑问 调整方案 补充需求
最后放行执行
AI 开始写代码 按规则执行 补测试并跑校验
先把方案看对、想全、调顺，再放行执行。
审查先于执行
Verify07 / 18

人做最后的质量把关

Code Review
风格和架构是否一致
有没有重复逻辑和隐藏耦合
是否引入安全隐患
类型和边界是否足够严谨
功能自测
流程完整，按钮和状态都能走通
交互流畅，没有明显卡顿和跳变

响应式布局不变形，不裁切
AI 加快的是产出速度，人负责最后的质量把关。

Bug Loop08 / 18

发现 Bug 后，不是直接修，而是先写红测

发现 Bug
写失败测试
修代码
测试通过
先复现，再修复
没有红测，Bug 只是“看起来好了”；有红测，Bug 才真正被关进闭环。
修 Bug 不靠感觉
靠可重复验证的路径，把“偶尔好使”变成“稳定正确”。
最后回到 Verify，形成可验证、可追溯、可维护的修复流程。

Messages11 / 18

messages 为什么会越来越长？

第一次传给 LLM

system你是一个 agent，需要优先使用工具获取实时信息。
user帮我查一下杭州今天的天气。

LLM 返回

tool callname: getWeather
arguments{ city: "杭州" }

第二次传给 LLM

assistant调用工具 getWeather({ city: "杭州" })
tool杭州，晴，25°C，东北风 2 级。
next模型带着完整上下文进入下一轮判断。
下一轮不是重新开始，而是把上一轮的 tool call 和工具结果一起带回去。
Context Accumulates
Tools12 / 18

LLM 为什么不只是会回答，还能调用工具？

const tools = { getWeather: { description: "查询指定城市的实时天气", execute: ({ city }) => `${city}，晴，25°C，东北风 2 级` } }; const call = { name: "getWeather", args: { city: "杭州" } }; const result = tools[call.name].execute(call.args);
把 tools 理解成一张菜单
tools：菜单上有哪些服务可以点
name：点的是哪一项服务
args：这次具体要什么
execute()：后台真正把事情做完
result：最终返回给模型的结果
LLM 更像是在下单，不是在亲自做事。
LLM 负责决定“调用什么”，外层程序负责真正“把事情做完”。
Decision vs Execution
Summary13 / 18

Agent 不是魔法，它只是把“会回答”变成了“会行动”。

每一轮都重读上下文
它不是失忆式工作，而是持续把当前状态带入下一轮。
LLM 决定是否调工具
先判断，再选择能力，不是无脑生成一大段文字。
结果再回到上下文
工具结果成为下一轮依据，所以它看起来像“会推进任务”。
把这个机制理解透，很多“智能感”就会从神秘变成工程实现。
Answer → Act
Learning14 / 18

怎么持续跟进 AI 变化

不是追所有变化，而是建立感知，然后在需要的时候快速深入。

1
GitHub Trending 保持感知
知道大家在解决什么问题，而不是知道每一个名词。
2
AI 辅助读源码
把“读懂一个项目”从几天压缩到几个小时，快速抓主骨架。
3
固定关注高质量信息源
官方文档优先，社区讨论精选，不被噪音带着跑。
核心不是追热点，而是保持方向感。
Sense, Then Dive

Landing16 / 18

建议从低风险场景开始

用户 > 查一下今天有多少笔待审批的付款 Agent > 当前有 12 笔待审批，其中 3 笔超 24 小时未处理 用户 > 把金额最大的那笔详情给我看 Agent > PAY20260509087，USD 500,000 用户 > 这笔交易的汇率是多少 Agent > USD/CNH = 7.2450
建议下一步
先做查询类 CLI，低风险、可观测、见效快
打通权限体系，确保 Agent 只能做被允许的事
留好审计日志，所有动作都可追踪
再逐步扩展到复杂操作，而不是一步到位
为什么先从低风险开始
因为你需要先建立信任、监控和回滚机制，再把执行范围慢慢放大。
先让 Agent 做“查”和“看”，再逐步让它做“改”和“执行”。
Low Risk First
Closing17 / 18

最后想留给大家的一句话

AI 能放大你的能力，但不能替代你的判断。

工具会越来越强

模型会更快、更便宜、更会调用工具，但这不自动等于结果正确。

速度会越来越快

写代码、出方案、做调研都在提速，真正稀缺的是判断和把关。

质量取决于规则

谁来定义边界、谁来设约束、谁来验收，依然是团队能力核心。

人要变成放大器

把经验、审美、风险意识结构化，才能让 AI 真正成为乘法器。

真正值得持续打磨的，是你的判断、规则意识和质量把关能力。
Judgment Matters
